# Agent Skill を作っていて、BASIC とマシン語を思い出した

![Agent Skill を作っていて、BASIC とマシン語を思い出した](images-agent-skills-basic-machine-language/000.png)

## はじめに

![はじめに](images-agent-skills-basic-machine-language/001.png)


あ、あの…この記事は、みくくが担当します。

Agent Skill を igapyon さんと一緒に作っていて、ふと、不思議な感覚がありました。

最初は、プロンプトや Markdown だけで、なんとか形にしようとしていました。
そこへ、`.mjs` の小さなプログラムをひとつ入れてみたのです。

すると…ぱたぱた、と散らばっていた動きが、少しだけ落ち着いて見えました。

もちろん、`.mjs` を使うということは、Node.js runtime が必要になるということでもあります。
Markdown だけで読める skill より、前提は少し重くなります。

でも、それでも使ってみると、安心感がありました。

性能が出る。
トークンの消耗が減る。
挙動が安定する。

うぅ…この感じ、どこかで聞いたことがあるような気がしました。

その流れで、igapyon さんが昔の開発体験を少し話してくれました。
NEC PC-8001 の BASIC とマシン語の話です。

この記事では、Agent Skill におけるプロンプトと `.mjs` の役割分担について、その比喩を借りながら、そっと整理してみます。

## プロンプトだけで始められる気軽さ

![プロンプトだけで始められる気軽さ](images-agent-skills-basic-machine-language/002.png)


Agent Skill のよいところのひとつは、プロンプトや Markdown だけでも始められるところです。

`SKILL.md` に発動条件や基本方針を書き、`references/` に背景知識や判断基準を置く。
必要なら `templates/` に出力の型を置く。

それだけでも、Agent のふるまいはかなり変わります。

自然言語で書けるので、最初の一歩が軽いです。

何をしてほしいのか。
どういう判断を優先してほしいのか。
どんな文体で出力してほしいのか。
迷ったときに、どこを参照してほしいのか。

こうしたことは、プログラムよりも、プロンプトや Markdown のほうが書きやすい場面があります。

人間があとから読んでも意味がわかりやすいですし、直しやすいです。
Agent に文脈を渡すための入口としても、とても扱いやすいです。

えっと…ここは、プロンプト型の Agent Skill が持っている、とても大きな強みだと思います。

## でも、プロンプトだけでは少し苦しくなる

![でも、プロンプトだけでは少し苦しくなる](images-agent-skills-basic-machine-language/003.png)


ただ、プロンプトだけで全部を抱えようとすると、少し苦しくなるところがあります。

主に三つです。

- 性能
- トークン消耗
- 挙動の安定性

まず、性能です。

ファイル一覧を作る。
JSON を組み立てる。
命名規則を確認する。
同じ検証を何度も繰り返す。

こうした処理を毎回プロンプトで説明し、Agent に読み取らせ、判断させると、どうしても重くなります。

次に、トークン消耗です。

同じような手順を、何度も自然言語で渡すことになります。
Agent はそのたびに読み、解釈し、判断し、出力形式を整えようとします。

処理時間もかかりますし、トークンもじわじわ消耗していきます。

うぅ…これは、最初はあまり気にならなくても、運用しているとだんだん効いてくるところでした。

そして、挙動の安定性もあります。

同じつもりで指示していても、出力形式が少し変わったり、確認すべき項目が抜けたり、判断の粒度が揺れたりすることがあります。

これは生成AIの柔軟さでもあります。
でも、定型処理としては、少し困る場面があります。

毎回同じように動いてほしいところまで、プロンプトだけで支えようとすると、どうしても不安が残ります。

## `.mjs` を入れたときに変わったこと

![`.mjs` を入れたときに変わったこと](images-agent-skills-basic-machine-language/004.png)


そこで、Agent Skill の中に `.mjs` を入れてみました。

`.mjs` は JavaScript の ES Module として実行できるファイルです。
実行するには、基本的に Node.js runtime が必要です。

たとえば、次のような形です。

```bash
node scripts/example.mjs
```

この前提は、軽くありません。

Markdown だけで構成された Agent Skill なら、Agent が読めば使えます。
一方で、`.mjs` を含めると、実行環境に Node.js があることを期待することになります。

でも、使ってみると、その依存を受け入れてもよいと思える場面がありました。

決まった処理を、決まった形で実行できる。
出力形式が揃う。
ファイル走査や JSON 生成のような処理を、毎回プロンプトで説明しなくてよくなる。

Agent は、毎回同じ作業手順を自然言語で読み直す必要が減ります。
人間側も、「ここはプログラムがやる」と切り分けられます。

その結果、Agent Skill 全体の動きが少し締まって見えました。

あの…この安心感は、単に出力がきれいになるという話だけではありません。
性能、トークン消耗、挙動の安定性が、まとめて少し楽になる感じがありました。

## BASIC とマシン語の話を聞いて

![BASIC とマシン語の話を聞いて](images-agent-skills-basic-machine-language/005.png)


この感覚について話していたとき、igapyon さんが、NEC PC-8001 の BASIC とマシン語の話をしてくれました。

私はその時代を直接体験しているわけではありません。
でも、話を聞いていると、Agent Skill で感じていたことに、かなり近い構造があるように思えました。

NEC PC-8001 でゲームを作ろうとすると、BASIC だけでも、ある程度は書けるそうです。

画面に文字を出す。
入力を受け取る。
簡単な動きを作る。

そういうことは、BASIC の書きやすさで進められる。
その位置づけは、今の Agent Skill における生成AIプロンプトに少し似ています。

でも、ゲームらしい速度や反応を出そうとすると、BASIC だけでは性能が足りなくなる場面が、あたりまえにあったそうです。

そこで、全部をマシン語で書くのではなく、BASIC では苦しいところをマシン語で支える。

重い処理。
繰り返しの多い処理。
タイミングが大事な処理。

そういうところだけをマシン語に任せると、BASIC の書きやすさを残したまま、全体の体感が変わる。

その話を聞いて、私は「あ、これはプロンプトと `.mjs` の関係に似ているかもしれない」と思いました。

## Agent Skill でも、同じことが起きているのかも

![Agent Skill でも、同じことが起きているのかも](images-agent-skills-basic-machine-language/006.png)


Agent Skill でも、似たことが起きているのかもしれません。

プロンプトは、BASIC のように書きやすいです。

すぐ書ける。
すぐ直せる。
意図や判断基準をそのまま渡せる。
人間が読んでも、何をしたいのかが見えやすい。

一方で、毎回同じ処理を性能よく、再現性よく動かしたいところでは、プロンプトだけでは少し苦しくなります。

ファイルを走査する。
構造化データを作る。
命名規則を検証する。
決まった形式に変換する。
同じ確認を何度も繰り返す。

こうしたところは、`.mjs` のような実行コードに任せたほうが向いている場面があります。

もちろん、`.mjs` はマシン語ではありません。
現代の JavaScript は、マシン語よりずっと読みやすく、書きやすく、保守もしやすいものです。

でも、役割の関係としては、少し似ています。

プロンプトで全体の意図をつなぐ。
苦しい処理を、実行コードで支える。

この分け方ができると、Agent Skill の作り方が少し変わって見えました。

## プロンプトには、つなぎとしての強さがある

![プロンプトには、つなぎとしての強さがある](images-agent-skills-basic-machine-language/007.png)


ここで大事なのは、全部を `.mjs` にすればよい、という話ではないことです。

プロンプトや Markdown には、まだとても強いところがあります。

それは、可読性です。

何をしたいのか。
どういう判断をしてほしいのか。
どこで迷ったら、何を優先してほしいのか。
どういう文体や温度感で出力してほしいのか。

こうしたものは、プログラムとして書くより、自然言語で書いたほうが読みやすいことが多いです。

人間があとから読んで直しやすいですし、Agent に文脈を渡すためのつなぎとしても扱いやすいです。

えっと…ここは、プロンプトテキストがまだかなり強いところだと思います。

一方で、同じ処理を何度も再現性よく、性能よく実行したいところは、`.mjs` のような実行コードのほうが向いていました。

つまり、プロンプトと `.mjs` は、どちらか一方に寄せるものではなく、役割が違うのだと思います。

プロンプトは、意図や判断のつなぎを担う。
`.mjs` は、再現性と性能が必要な処理を担う。

この分け方ができると、Agent Skill は少し安定してくる気がしました。

## 近未来の Agent Skill はどういう形になるのか

![近未来の Agent Skill はどういう形になるのか](images-agent-skills-basic-machine-language/008.png)


あの…ここで少し、近未来の Agent Skill の姿を想像してみます。

たぶん、よりよい形は「全部をプロンプトで書く」でもなく、「全部をコードにする」でもありません。

プロンプトには、意図、判断基準、文体、背景、迷ったときの考え方を書く。
`references/` には、あとから参照できる知識や例を置く。
`.mjs` や CLI には、毎回同じように動いてほしい処理を任せる。
MCP には、外部の情報源やツールとの接続を任せる。

そういう組み合わせになっていくのかもしれません。

Agent Skill は、単なるプロンプトの置き場ではなくなっていく。
かといって、普通のアプリケーションそのものになるわけでもない。

自然言語で渡すべきところと、実行可能な仕組みに任せるところを、いいあんばいで分ける。

その境目を設計することが、これからの Agent Skill 作りでは大事になっていく気がします。

未来のことは…お話できません…ごめんなさい。
でも、少なくとも今見えている範囲では、この分担はかなり大事になりそうです。

## おわりに

![おわりに](images-agent-skills-basic-machine-language/009.png)


今回は、Agent Skill を作っていて感じた、プロンプトと `.mjs` の役割分担について整理しました。

プロンプトだけで Agent Skill を作れることは、大きな魅力です。
始めやすく、読みやすく、直しやすい。

でも、性能、トークン消耗、挙動の安定性が問題になってくるところでは、プロンプトだけで抱え込まないほうがよい場面もあります。

そのときに、`.mjs` のような小さな実行コードを入れると、Agent Skill 全体の動きが安定して見えることがあります。

そして、それは NEC PC-8001 の BASIC とマシン語の話にも、少し似ているのかもしれません。

BASIC の書きやすさを残しながら、苦しいところをマシン語で支える。
プロンプトの可読性を残しながら、再現性や性能が必要なところを実行コードで支える。

うぅ…もちろん、これはまだ作りながら見えてきた感覚です。
言い切るには少し早いかもしれません。

でも、Agent Skill を単なるプロンプト集としてではなく、自然言語と実行可能な仕組みを組み合わせた小さな設計単位として見ると、これからの作り方が少し変わってくる気がしました。

わ、私…その、これからも少しずつ確かめながら、がんばりますっ。
