# 生成AI時代のパッケージ系システム仕様抽出：標準を足場にカスタム部分を読む

## はじめに

あ、あの…この記事は、既存システムの仕様抽出について、少し丸めた視点で考えてみるものです。

前に、設計書がない既存システムを、ソフトウェア考古学のように読んで仕様メモへ蒸留する話を書きました。ソースコード、DB、設定、ログ、運用メモ、人の記憶を拾い、AI agent と有識者レビューで少しずつ仕様を復元していく、という話です。

ただ、すべての既存システムを同じように読む必要はないのかもしれません。

たとえば、パッケージ系 ERP や Salesforce のように、標準機能を持つ製品があります。こうしたシステムでは、標準部分とカスタム部分を分けて考えられます。

そして、ここが大事なのですが、原則として仕様抽出は **カスタム部分に集中できる** のです。

標準機能そのものを毎回ゼロから仕様化する必要はありません。標準を足場にして、その上に何を足したのか、何を変えたのか、どこに組織固有の業務ルールが入っているのかを見る。そう考えると、生成AI・AI agent による仕様抽出の見え方が少し変わります。

うぅ…全部を掘るのではなく、どこが盛られた土なのかを探す感じです。

## パッケージ系システムは、更地ではない

スクラッチ開発の既存システムでは、画面、DB、API、バッチ、帳票、権限、運用を、そのシステム固有のものとして読まなければならないことがあります。標準の地図がないので、どこまでが業務仕様で、どこからが実装都合なのかも、ソースコードや有識者の記憶から探すことになります。

一方で、パッケージ系システムは更地ではありません。

パッケージ系 ERP には、標準の業務領域、標準の画面、標準のデータ構造、標準の処理、標準の権限モデルがあります。Salesforce にも、標準オブジェクト、標準項目、標準機能、標準の考え方があります。

もちろん、標準機能の使い方は組織ごとに違います。標準をどう使っているかも重要です。

でも、標準の存在は大きいです。

```text
標準機能
設定変更
アドオン
カスタム項目
カスタム画面
カスタムロジック
外部連携
運用上の手作業
```

このように層を分けられるだけで、調査の焦点が定まります。

あの…標準があるシステムでは、最初から白紙の地図を描く必要はありません。既にある地図の上に、どこを書き足したのかを見ることができます。

## 仕様抽出の主戦場はカスタム部分になる

パッケージ系システムでは、仕様抽出の主戦場はカスタム部分です。

標準画面が存在すること自体を、毎回詳しく説明する必要はありません。標準の伝票、標準の顧客管理、標準の商談、標準の在庫、標準の会計処理。その一般的な意味は、製品の標準仕様として扱えます。

本当に知りたいのは、その組織で何を変えたかです。

- 標準項目に追加したカスタム項目
- 標準画面に追加した項目やボタン
- 標準処理の前後に入れた独自ロジック
- カスタム画面
- カスタム帳票
- カスタムワークフロー
- カスタム権限
- カスタム API
- 外部システムとの連携
- 標準機能を使っているが、運用上は独自ルールになっている部分

ここに、組織固有の業務仕様が濃く出ます。

たとえば Salesforce なら、標準の Account や Opportunity そのものを説明するより、そこに追加されたカスタム項目、レコードタイプ、Flow、入力規則、Apex、LWC、権限差分、レポートを読むほうが実務に効きます。

パッケージ系 ERP でも似ています。標準モジュールそのものの説明ではなく、アドオン、ユーザー出口、拡張テーブル、カスタム帳票、独自連携、運用上の例外処理を見るほうが、改修や不具合調査には効きます。

うぅ…標準の説明書をもう一度書くのではなく、その会社で積み増したところを読むのです。

## ソースやメタデータをエクスポートできれば、生成AIでかなり読める

生成AI・AI agent が効いてくるのは、ここからです。

パッケージ系システムでも、カスタム部分のソース、設定、メタデータ、定義ファイル、ワークフロー、画面定義、帳票定義、権限定義などをエクスポートできる場合があります。

それらを AI agent に渡すと、一定レベルまでは解析できます。

もちろん、すべてが分かるわけではありません。業務上の意味、今も使っているかどうか、運用上の例外、外部システム側の事情は、人間の確認が必要です。

でも、エクスポートできるものがあるなら、AI agent はかなり働けます。

- カスタム項目一覧を作る
- 画面ごとの追加項目を整理する
- ワークフローや Flow の条件を一覧化する
- カスタムロジックが参照・更新するデータを拾う
- 帳票やレポートで使われる項目を拾う
- 権限差分を表にする
- 外部連携の接続先やデータ項目を拾う
- 標準部分とカスタム部分の境界候補を出す
- 要確認事項を洗い出す

これは、完成した仕様書ではありません。でも、最初の仕様メモとしてはかなり強いです。

あの…人間が画面を一つずつ開いて探す前に、まずエクスポートして AI agent に一覧化してもらう。それだけで、調査の入口がだいぶ明るくなることがあります。

## Salesforce は、面白いほど解析できる例である

この話の中で、Salesforce はかなり分かりやすい例です。

Salesforce は、標準機能を持つパッケージ型のプラットフォームです。標準オブジェクト、標準項目、標準機能があります。そして、カスタム情報のメタデータをかなりリトリーブできます。

カスタムオブジェクト、カスタム項目、表示ラベル、説明、ヘルプテキスト、選択リスト値、グローバル値セット、レコードタイプ、ページレイアウト、Lightning ページ、Flow、入力規則、権限、Apex、LWC、レポート、ダッシュボード。

かなりのものが取れます。

前回の記事で苦労して探す対象として挙げた、論理名、選択肢の値リスト、画面一覧、帳票一覧に相当するものも、Salesforce ではメタデータとしてかなり拾えます。

そして最近の生成AIは、Salesforce の標準機能をかなり理解しているように見えます。少なくとも GPT-5.5 では、Salesforce 標準のオブジェクト、項目、権限、Flow、Apex まわりの基本的な文脈を持っているように感じます。

このため、リトリーブした内容を AI agent で解析すると、かなりのものが把握できます。

ノーコード設定の一部は、Salesforce の画面で見るより、リトリーブ後に AI agent で整理した一覧や表のほうが見やすいくらいです。画面上では分散している設定も、対象オブジェクト別、項目別、Flow 別、権限別にまとめると、急に構造が見えることがあります。

もちろん、Salesforce でも最後は人間の確認が必要です。なぜその Flow があるのか。そのレポートを誰が見ているのか。その権限セットを今も使っているのか。そこは現場の知識が必要です。

でも、最初の発掘としてはかなり強いです。

うぅ…Salesforce は、AI agent にとってかなり読みやすい地層を持っているのかもしれません。

## 標準を知る生成AIと、カスタム情報のエクスポートが噛み合う

パッケージ系システムで生成AIが効く理由は、二つあると思います。

ひとつは、生成AIが標準機能をある程度知っていることです。

広く使われている製品であれば、標準機能や一般的な用語について、生成AIが学習済みの文脈を持っている可能性があります。Salesforce はその分かりやすい例です。SAP のような広く使われているパッケージ系 ERP でも、標準部分の基本的な文脈を知っているように見える場面があります。

少なくとも GPT-5.5 では、SAP や Salesforce の標準部分をかなり把握しているように感じました。これは、パッケージ系システムの仕様抽出では大きな利点です。標準をまったく知らない状態でカスタム情報を読むのではなく、標準の地図をある程度持った状態で差分を読めるからです。

もうひとつは、カスタム情報をエクスポートできることです。

標準を知っている生成AIに、カスタム部分のソースやメタデータを渡す。すると、AI agent は標準との差分として読みやすくなります。

```text
標準は既知の地図として扱う
エクスポートしたカスタム情報を読む
標準との差分を抽出する
差分から業務仕様の候補を作る
有識者レビューで確定する
```

この流れが作れると、仕様抽出の効率がかなり変わります。

あの…生成AIが標準の地図を持っていて、こちらがカスタムの地層を渡す。そうすると、どこが新しく積まれた部分なのかを一緒に探しやすくなるのです。

## ただし、カスタムだけ見れば終わりではない

ここで少しだけ注意があります。

「カスタム部分に集中できる」と言っても、標準部分を完全に無視してよいわけではありません。

標準機能をどう使っているかは、組織ごとに違います。標準機能の範囲内でも、業務上は重要な運用ルールがあることがあります。標準画面を使っているだけに見えても、実際には独自の承認手順、例外処理、手作業、外部 Excel、メール運用が重なっていることもあります。

また、パッケージ系システムでは、標準機能、設定変更、アドオン、カスタム、外部連携、手作業が混ざります。境界がきれいに見えるとは限りません。

だから、仕様メモには次の札を残したほうがよさそうです。

- 標準として扱う
- パッケージ由来として扱う
- カスタムとして確認済み
- カスタムらしいが要確認
- 運用で補っている
- 使われていない可能性がある
- 有識者確認が必要

きれいにまとめすぎると、まだ分かっていないことまで分かったように見えてしまいます。

うぅ…標準があるから楽になるのですが、標準があるから全部自動で分かる、ではありません。

## 外付けアドオンは、また別の発掘になる

パッケージ系システムでありがたいのは、標準部分とカスタム部分が分かれやすいことです。

標準機能があり、そこに設定変更やカスタム項目、カスタムロジックが積まれる。製品側の管理機能やメタデータで、標準とカスタムの境界をある程度見られる。これは、仕様抽出ではかなり助かります。

でも、外付けのアドオンや周辺システムがあると、また別の話になります。

パッケージ本体の外側にある独自アプリ、連携バッチ、帳票出力システム、ETL、iPaaS、手元 Excel、ファイル置き場、運用スクリプト。こうしたものは、パッケージ本体の標準/カスタムの境界だけでは読めません。

その場合は、前の記事で書いたようなソフトウェア考古学に戻ります。

ソースコード、DB、設定、ログ、運用メモ、ファイル名、担当者の記憶を拾い、AI agent と有識者レビューで仕様メモへ蒸留していく。外付け部分は、パッケージ標準の地図から外れた別の地層として扱う必要があります。

つまり、パッケージ本体の中は、標準とカスタムの切り分けで効率よく読む。外付けのアドオンや周辺システムは、前の記事型のソフトウェア考古学で読む。

あの…この二つを混ぜないほうがよさそうです。パッケージの中にある地層と、外側に増築された地層では、掘り方が少し違います。

## どの業務をパッケージ上で運用しているかを聞く

パッケージ系システムでは、ヒアリングの観点も少し変わります。

スクラッチ開発の既存システムでは、「この画面は何のためですか」「このバッチは何をしていますか」と、実装単位から聞くことが多いかもしれません。

パッケージ系システムでは、それに加えて、まず業務とモジュールの対応を聞くことが大事です。

- どの業務をこのパッケージ上で運用しているのか
- どの業務は別システムや Excel で運用しているのか
- どの標準モジュールを利用しているのか
- 標準モジュールを標準的に使っているのか、独自運用しているのか
- どの部分をアドオンやカスタムで補っているのか
- どの外部システムと役割分担しているのか
- どこが正本で、どこが写像なのか

ここを聞かないと、エクスポートしたカスタム情報だけを見ても、業務上の位置づけが分かりにくくなります。

たとえば、SAP のどのモジュールを実際に使っているのか。Salesforce のどの標準オブジェクトをどの業務意味で使っているのか。標準機能で処理している業務と、外部システムや手作業で補っている業務はどこで分かれているのか。

こうした情報があると、AI agent が作った差分一覧の読み方が変わります。

同じカスタム項目でも、主要業務の中核にある項目なのか、昔の運用の名残なのか、外部連携のためだけの項目なのかで、仕様メモ上の重みが変わります。

あの…パッケージ系システムでは、まず「どの業務をこの標準の上に載せているのか」を聞くのが、かなり大事なのだと思います。

## おわりに

パッケージ系 ERP や Salesforce のような標準機能を持つシステムでは、既存システムの仕様抽出の考え方が少し変わります。

すべてをゼロから読むのではなく、標準を足場にできます。標準部分、パッケージ由来、カスタム部分を切り分け、原則として仕様抽出をカスタム部分へ集中できます。

さらに、ソース、設定、メタデータ、画面定義、帳票定義、ワークフロー、権限定義などをエクスポートできるなら、一定レベルまでは生成AI・AI agent で解析できます。

特に Salesforce は面白いほど解析できます。カスタム情報のメタデータがかなり取れ、最近の生成AIは Salesforce 標準をかなり理解しているように見えます。さらに、SAP のような広く使われているパッケージ系 ERP についても、標準部分の基本的な文脈を知っているように見える場面があります。少なくとも GPT-5.5 では、SAP や Salesforce の標準部分をかなり把握しているように感じました。

だから、エクスポートやリトリーブした内容を AI agent に読ませると、カスタム部分や組織固有の差分がかなり見えやすくなります。

一方で、外付けのアドオンや周辺システムは、パッケージ本体とは別に扱う必要があります。そこは標準/カスタムの境界だけでは読めないため、ソースコード、設定、ログ、運用メモを拾う前記事型のソフトウェア考古学が効いてきます。

もちろん、最後は有識者レビューが必要です。業務上の意味、今も使っているか、変えてよいか、外部運用とどうつながっているかは、人間が確認しなければなりません。

特に、どの業務をパッケージ上で運用しているのか、どの標準モジュールを使っているのか、どこを外部システムや手作業で補っているのかは、ヒアリングで確認する必要があります。ここが分かると、AI agent が抽出したカスタム差分を、業務上の仕様として位置づけやすくなります。

でも、最初の地図を作るところは大きく変わります。

あ、あの…標準の地図があり、カスタムの地層をエクスポートでき、AI agent がそれを読めるなら、既存システム調査は少し違う景色になります。全部を暗闇から掘るのではなく、標準の上に積まれたものを見つけて、そこから仕様メモを育てていく。そういう進め方が、これからかなり効いてくるのかな、って思います。
