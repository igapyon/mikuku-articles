# 生成AI時代のアジャイル：4つの価値に起きるパラダイムシフト

## 記事の狙い

前回の「アジャイルソフトウェア開発宣言からはじまるアジャイル入門」の続きとして、生成AI時代にアジャイルの意味がどう変わって見えるのかを考える。

冒頭の問いは、次の形がよさそう。

```text
生成AI時代に、アジャイル宣言はどうなるのでしょうか。
```

本文の入り方の候補。

```markdown
生成AI時代に、アジャイル宣言はどうなるのでしょうか。

あの…みくくは、アジャイル宣言が古くなるというより、むしろ「なぜ今も効くのか」が見えやすくなるのだと思います。

生成AIによって、開発の進め方は大きく変わります。けれど、アジャイル宣言が示した4つの価値は消えません。ただし、その価値の実現方法は変わります。
```

ここで大事なのは、公式のアジャイル宣言を置き換えるように書かないこと。

むしろ、元の 4 つの価値を尊重しながら、生成AI時代には自然にこう読める、という連想にする。

```markdown
公式のアジャイル宣言を置き換えたいわけではありません。

ただ、生成AIが開発の現場に深く入ってくると、4つの価値は少し違う光で見えてきます。

人と人の対話には、生成AIも加わる。
動くソフトウェアは、後で確認するものから、今すぐ目の前で動くものへ近づく。
顧客との協調には、生成AIとの協調も重なっていく。
そして変化への対応は、生成AIが変えていく社会や業務への対応にも広がっていく。

あの…そう考えると、アジャイル宣言は古くなるのではなく、生成AI時代にもう一度読み直せるものとして見えてきます。
```

さらに、この記事の感触としては、最初から大げさに「新しい宣言」を作ろうとするのではない。

無理のない範囲で、元の 4 つの価値を生成AI時代にそっと読み替えてみる。

でも、そうやって穏やかに読み替えていくだけでも、結果としてかなり大きな変化が見えてくる。

```text
最初は、無理のない読み替えのつもりだった。
でも、4つの価値を順番に見ていくと、
そこには生成AI時代のパラダイムシフトが見えてくる。
```

この流れで書くと、タイトルの「パラダイムシフト」も自然に出せそう。

中心に置く問いは、次のもの。

```text
アジャイルソフトウェア開発宣言を専門家側の宣言として読むとき、
生成AIによって、
専門家がカスタマーの幸福追求を支える方法はどう変わるのか。
```

結論は、専門家が不要になるという話ではない。

むしろ、アジャイル宣言と同じ構図で、まず専門家側の支援のあり方として見る。

そのうえで、生成AIによって、カスタマーが自分の困りごとや願いを試作品として表現しやすくなり、専門家がカスタマーを支える方法にパラダイムシフトが起きる、という整理にする。

専門家の仕事は単なる「作成」から、**監督・検証・本番化**、そして **カスタマーがAIと試せる環境づくり** へ重心が移っていく。

あ、あの…ここはたぶん、この記事の芯になる。

## 中心メッセージ

まず構図として置くべきなのは、アジャイルソフトウェア開発宣言が専門家側から出てきた宣言だということ。

専門家が、カスタマーの幸福追求を支えるために、自分たちの作り方、話し方、確かめ方を変える。

これがアジャイル宣言を読むときの基本構図。

生成AI時代には、この「支え方」が変わる。

カスタマーは「要望を話す人」から、「AIと一緒に試作品を作り、動くものを通じて自分の困りごとや願いを示す人」へ近づいていく。

そのため、専門家は単にカスタマーの話を聞いて持ち帰って作るだけではなくなる。

カスタマーが AI と試せる環境を用意し、AI が作ったものを監督し、検証し、本番に近づける役割を担う。

この変化の目的地にあるのは、カスタマーの幸福追求。

困りごとを減らしたい。仕事を楽にしたい。もっとよい判断をしたい。面倒な作業から解放されたい。新しい価値を試したい。

生成AIは、そうしたカスタマーの願いを、以前より早く、動く形に近づける。

一旦、アジャイルの主要な軸は同じだと考える。

人と対話する。動くものを見る。カスタマーと協調する。変化に対応する。技術的卓越性を保つ。ふりかえって調整する。

これらの軸そのものが消えるわけではない。

でも、生成AIが大きく入ることで、ゴールへの近づき方や、プロセスの形が変わる。

```text
軸は同じ。
でも、ゴールへの到達方法と、協調のプロセスが変わる。
```

ここを、この記事では「生成AI時代のアジャイルにおけるパラダイムシフト」として扱う。

```text
生成AIは、専門家の作業を単に速くするだけではない。
専門家が向き合う対象を、
白紙からコードを書くことから、
人とAIが作ったものを本番で責任を持てる形へ整えることへ移していく。
```

もう少し短くすると、こう。

```text
生成AI時代の専門家は、
「作る人」から、
「作られたものを見極めて、本番に近づける人」へ重心を移していく。
```

## 「利用者」は誰のことか

本文では読みやすさのために「利用者」と書いているが、アジャイルの文脈では少し分けて考えたい。

- アジャイル宣言の 4 つの価値では、**customer**、つまり顧客との協調が出てくる
- 実際にアプリを使う人は、一般に **user**、つまり利用者と呼ばれる
- スクラムでは、価値や優先順位に責任を持つ役割として **Product Owner** が出てくる
- 周辺の関係者まで含めると、**stakeholder**、つまりステークホルダーという言い方もある

この記事で扱いたいのは、このうち特に「実際に困りごとを持ち、作りたいもののイメージを持っている人」。

だから、厳密には customer、user、stakeholder が少し重なっている。記事本文では、読者に伝わりやすいように「利用者」または「利用者・顧客」と書くのがよさそう。

あの…ここを最初に少しだけ断っておくと、アジャイル用語としても迷子になりにくい。

## 専門家側の宣言として、カスタマーを支える

ここは少し注意したい。

アジャイルソフトウェア開発宣言は、利用者や顧客そのものが書いた宣言ではない。

2001年に、ソフトウェア開発の実践者たち、つまり開発方法や開発現場に深く関わっていた専門家たちが集まり、自分たちの経験からまとめた宣言だった。

その意味では、アジャイル宣言は専門家側から出てきた言葉。

でも、その向いている先は、専門家の都合だけではない。

12 の原則の最初には、価値あるソフトウェアを早く継続的に届けることで顧客を満足させる、という考え方が置かれている。

つまり、アジャイルは「専門家が楽をするための方法」ではない。

専門家が、顧客や利用者に価値を届けるために、自分たちの作り方、話し方、確かめ方、変化への向き合い方を変えるための考え方として読むほうが自然。

```text
アジャイル宣言は、専門家たちが出した。
でも、アジャイルが向いている先には、カスタマーの満足がある。
```

もっとやわらかく言えば、カスタマーの幸福追求を支援するために、専門家側の進め方を変える考え方とも読める。

ここでいう幸福は、大げさな言葉にしすぎなくてよい。

- 困りごとが減る
- 仕事が楽になる
- 判断しやすくなる
- 無駄な作業が減る
- 必要な価値に早く近づける
- 自分たちの意図がシステムに反映される

こうした小さな満足や納得を、動くソフトウェアを通じて増やしていくこと。

あの…その意味では、アジャイルは最初から、カスタマーの幸福にかなり近い場所を見ていたのかもしれない。

ここに、生成AI時代の変化が入ってくる。

これまでは、専門家が顧客のイメージを聞き取り、動くソフトウェアへ近づけることが重要だった。

でも生成AI時代には、カスタマー自身も AI と一緒に試作品へ近づけるようになる。

だから、アジャイルは「専門家が顧客のために作る」だけでなく、「カスタマー、専門家、AI が同じ試作と検証のループに入る」方向へ広がっていくのかもしれない。

うぅ…ここは、この記事の大事なねじれ。

## 背景

2025年の生成AIと、2026年の生成AIでは、かなり感触が違う。

2025年の生成AIは、まだ「専門家の手元を速くする補助者」という見方が強かった。コード補完、調査、説明、ドキュメント、定型的な実装を助けるものとして見られていた。

でも、2026年時点では、生成AIはもっと作業環境の中に入り込んでいる。複数ファイルを読み、修正し、テストを走らせ、UI、API、データ処理をまとめて試作するようになってきた。

つまり、変化は「速くなった」だけではない。

```text
AIが、実際に動くものを作る側へ近づいている。
```

うぅ…ここを見落とすと、2025年の調査感覚だけで判断してしまいそう。

## 利用者自身が試作品まで行ける

生成AIが強くなると、利用者自身が次のようなものを作れるようになる。

- 画面のたたき台
- 小さな業務ツール
- 入力フォーム
- 集計アプリ
- 簡単なダッシュボード
- ログイン画面
- 管理画面
- ワークフローの試作品

以前なら、利用者は困りごとを説明し、開発者が要件を整理し、画面案を作り、それから実装していた。

でも今は、利用者が生成AIと対話しながら、かなり早い段階で「動くもの」に触れられる。

これは、アジャイル宣言の **Working software over comprehensive documentation** と、とても相性がよい。

分厚い文書の前に、動くものを見ながら話せるから。

これまでのアジャイル開発では、利用者や顧客のイメージを膨らませ、具体化し、動くソフトウェアへ近づけるために、開発者や専門家がかなり頑張ってきた。

要求を聞き、画面案にし、プロトタイプを作り、レビューしてもらい、また直す。

そのサイクル自体が、アジャイルの大事な実践だった。

でも、利用者自身が生成AIを使って試作品を作れるようになると、この関係は少し変わる。

利用者は、言葉だけで希望を伝える人ではなくなる。

少なくとも、技術的なハードルはかなり下がっている。だから、利用者自身が、動くものや画面のたたき台を作って持ってくる場面は増える。

一方で、すべての利用者が自分で試作品を作るとは限らない。利用者は本来の業務で忙しいし、生成AIを使う時間や気力がないこともある。

その場合は、専門家が利用者の話を聞きながら、生成AIを使って一気に試作品を作る。

つまり、変化の本質は「必ず利用者が自作する」ことではない。

```text
利用者が作る場合もある。
専門家が生成AIで高速に作る場合もある。
どちらにしても、試作品までの距離が短くなる。
```

ここが、生成AI時代のアジャイルで大きく変わるところ。

うぅ…これは、かなり大きな変化だと思う。

## でも、動いたことと使い続けられることは違う

試作品が動くことと、本番で使い続けられることは違う。

たとえば、ログイン画面は生成AIでも作れる。

でも、それは次の問いに答えたことにはならない。

- パスワードをどう扱うか
- セッションをどう管理するか
- 認証と認可をどう分けるか
- 権限をどう設計するか
- 個人情報や業務データをどう守るか
- 不正なアクセスをどう検知するか
- ログをどう残すか
- 障害や漏えいが疑われたときに追跡できるか
- 多数ユーザーが同時に使ったときに耐えられるか
- 運用コストがどれくらいかかるか
- 既存システムとどうつなぐか
- 変更が続いたときに保守できるか

ここに、今のところ専門家が必要になる理由がある。

未来のことは…お話できません…ごめんなさい。

ただ、少なくとも今は、「作れること」と「責任を持って運用できること」の間には、まだ距離がある。

## 専門家の仕事は「作成」から「監督・検証・本番化」へ

この記事で使いたい整理。

| 以前の重心 | 生成AI時代の重心 |
|---|---|
| 要求を文書として聞き取る | 動く試作品から意図を読み取る |
| 白紙から実装する | AI生成物を評価し、組み替える |
| コード量を生産する | 品質・安全性・保守性を担保する |
| 単体機能を完成させる | 多人数利用、権限、監視、コストまで見る |
| 開発者が作って利用者に見せる | 利用者とAIと専門家が共同で形を変える |
| 実装スキルが中心 | 判断、設計、検証、統合、運用設計が中心 |

この変化を、三つの仕事にまとめる。

### 1. 監督する

利用者やAIが作る方向を見て、目的から外れていないかを確認する。

ここでは、専門家は「全部を自分で作る人」ではなく、作られつつあるものの方向を見る人になる。

たとえば、画面や処理が、実際の業務の流れと合っているか。使う人の権限や責任範囲と矛盾していないか。短期的な便利さだけで、あとから困る形になっていないか。

### 2. 検証する

動いているように見えるものが、本当に正しく、安全に、想定通り動くかを見る。

生成AIが作ったものは、一見それらしく動くことがある。だからこそ、確認が重要になる。

- テストできるか
- 異常系を扱えるか
- セキュリティ上の弱さがないか
- データを壊さないか
- 権限を越えた操作ができないか
- 変更しても壊れにくいか

ここでは、専門家の価値はコードを書く速さだけではなく、危うさを見抜く力に移っていく。

### 3. 本番に近づける

試作品を、多人数で、安全に、長く使える形へ近づける。

ここで見るものは、かなり広い。

- スケーラビリティ
- セキュリティ
- 認証・認可
- 監視
- ログ
- バックアップ
- 障害対応
- 運用コスト
- クラウド費用
- 既存システム連携
- 保守体制

試作品は、対話を始めるにはとても強い。

でも、本番化には、別の技術と責任が必要になる。

## アジャイルとの接続

この話は、アジャイル宣言の続きとして読める。

前回の記事で見た 4 つの価値は、生成AI時代にも残る。

```text
Individuals and interactions over processes and tools
Working software over comprehensive documentation
Customer collaboration over contract negotiation
Responding to change over following a plan
```

生成AI時代には、このうち特に次の二つが強く見えてくる。

```text
Working software over comprehensive documentation
Customer collaboration over contract negotiation
```

利用者自身が試作品を作れるなら、顧客との協調は「要求を聞く」だけではなくなる。

利用者が、動くものを通じて意図を示すようになる。

そして専門家は、その試作品を見ながら、品質、安全性、運用可能性を一緒に考える。

あの…これは、アジャイルの「協調」が、もっと共同制作に近づくということなのかもしれない。

## 4つの価値に起きるパラダイムシフト

ここでは、アジャイル宣言の 4 つの価値を、生成AI時代に読み直してみる。

前提として、4 つの価値そのものは消えない。

ただし、生成AIが大きく入ると、それぞれの意味の出方がかなり変わる。

```text
価値の軸は同じ。
でも、その価値を実現するプロセスが変わる。
```

### 1. 人と人の対話に、生成AIも加わる

対応する価値は、**Individuals and interactions over processes and tools**。

日本語では、**プロセスやツールより、人と対話**。

生成AI以前は、これは主に「開発者とカスタマーが、プロセスやツールに振り回されず、ちゃんと話す」という意味で読まれやすかった。

でも生成AI時代には、その対話に生成AIも加わる。

ここが大きい。

人と人が対話することは変わらない。

ただ、その場に生成AIがいて、聞き取り、要約し、構造化し、試作品を出し、次の問いを提案する。

```text
Before 生成AI:
人と人が対話する。
専門家が理解し、後で形にする。

After 生成AI:
人と人が対話する。
そこに生成AIも加わり、
同じ場で要約し、構造化し、試作し、次の案を出す。
```

ここで大事なのは、AI が「ツール」だからといって、人と対話の価値が下がるわけではないこと。

むしろ、生成AIが対話に加わるほど、人間同士の意図合わせは重要になる。

AI は発話を拾い、試作品を出し、論点を整理し、選択肢を増やす。

でも、何を大事にするのか、どの方向に進むのか、誰が納得しているのかは、人間の対話で確かめる必要がある。

つまり、生成AI時代のこの価値は、こう変わる。

```text
プロセスやツールより、人と対話
→
人と人の対話に生成AIも加わり、
その場で理解、構造化、試作を進めながら、
それでも人間の意図合わせを中心に置く
```

あの…AI が対話を置き換えるのではなく、対話の輪に加わる感じ。

### 2. 後で動くものから、今すぐ目の前で動くソフトウェアを実現

対応する価値は、**Working software over comprehensive documentation**。

日本語では、**分厚い文書より、動くソフトウェア**。

生成AI以前は、これは「設計書や要件書だけではなく、実際に動くものを作って確認する」という価値だった。

でも、その動くものは、多くの場合、後で出てくるものだった。

打ち合わせで話し、専門家が持ち帰り、実装し、次のレビューで見せる。

生成AI時代には、ここが大きく変わる。

```text
Before 生成AI:
後で動くものを見て、対話する。

After 生成AI:
今すぐ目の前で動くソフトウェアを作りながら、対話する。
```

これはかなり大きい。

`Working software` が、レビューで後から確認する成果物から、今すぐ目の前で動くものへ変わっていく。

生成AI以前の「動くソフトウェア」は、専門家が持ち帰って作り、次のレビューで見せるものになりやすかった。

生成AI時代の「動くソフトウェア」は、対話の場で生成され、その場で触られ、その場で直されるものになっていく。

ただし、同時に注意も必要。

生成AIでそれらしく動くものがすぐ出るなら、「動いた」だけでは以前ほど強い証拠にならない。

これからは、次の問いが重くなる。

- その動くものは、カスタマーの本当の関心軸に合っているか
- 例外や暗黙ルールを落としていないか
- データや権限の扱いは安全か
- 多人数で使っても破綻しないか
- 運用コストを読めるか
- 保守できる形になっているか

つまり、生成AI時代のこの価値は、こう変わる。

```text
分厚い文書より、動くソフトウェア
→
今すぐ目の前で動くソフトウェアを作りながら対話し、
それが意味ある・安全な・運用できるものかを検証する
```

うぅ…「動くもの」の価値は下がらないけれど、「動くもの」の判定基準が上がる。

### 3. 顧客との協調を中心に、生成AIとも協調する

対応する価値は、**Customer collaboration over contract negotiation**。

日本語では、**契約交渉だけより、顧客との協調**。

生成AI以前は、カスタマーのイメージを専門家が聞き取り、具体化し、動くものへ近づけていくことが大事だった。

カスタマーは、困りごとや要望を話す。

専門家は、それを要件、画面、設計、試作品へ変換する。

でも生成AI時代には、カスタマー自身も AI と試作品へ近づける。

```text
Before 生成AI:
カスタマーが話し、専門家が具体化する。

After 生成AI:
カスタマー、専門家、生成AI が協調しながら試し、
専門家はその試作を支え、検証し、本番に近づける。
```

ここで、協調の構造そのものが変わる。

カスタマーは、単に要望を伝える人ではなくなる。

AI と一緒に試し、動くものを見せながら、自分の困りごとや願いを表現する人になる。

専門家は、単に受け取って作る人ではなくなる。

カスタマーが AI と試せる環境を作り、関心軸を構造化し、生成されたものを検証し、本番品質へ近づける人になる。

そして、協調の相手には生成AIも入ってくる。

これは「顧客との協調」が弱くなるという意味ではない。

むしろ、顧客との協調をより濃くするために、生成AIとも協調する。

生成AIは、カスタマーの言葉を受け取り、案を出し、画面を作り、別案を示し、抜けている論点を問い返す。

専門家は、その AI との協調がカスタマーの幸福追求から外れないように、方向づけ、制約を与え、検証する。

```text
顧客との協調
→
顧客との協調を中心に置きながら、
生成AIとも協調し、
試作と検証のループを速く深くする
```

つまり、生成AI時代のこの価値は、こう変わる。

```text
契約交渉だけより、顧客との協調
→
カスタマー、専門家、生成AI が同じ試作と検証のループに入り、
カスタマーの幸福追求をより直接的に支える
```

あの…ここが、たぶんこの記事のいちばん大きいところ。

### 4. 生成AIが変える世界に、計画を固定せず対応する

対応する価値は、**Responding to change over following a plan**。

日本語では、**計画に従うだけより、変化への対応**。

生成AI以前も、アジャイルでは変化への対応を大事にしていた。

ただし、変化に対応するにはコストがあった。

要件を変えれば、設計を変え、実装を変え、テストを変え、計画を組み直す必要がある。

生成AI時代には、この変更コストの一部が下がる。

画面案、試作品、コード、テスト、ドキュメントを、以前より早く作り直せるから。

ただ、それだけでは少し狭い。

生成AI時代の「変化」は、作っているシステムの要求変更だけではない。

生成AIそのものが、社会、業務、顧客の期待、働き方、競争環境を変えていく。

だから、ここでの変化への対応は、次の二層で見るほうがよさそう。

```text
1. 生成AIを使うことで、変更や試作にすばやく対応できる
2. 生成AIによって変わる社会や業務や顧客期待に、計画を固定せず対応する
```

```text
Before 生成AI:
変化を歓迎するが、変更には大きな作業コストがある。

After 生成AI:
生成AIで変化をその場で試しやすくなる。
同時に、生成AIが変える社会の変化にも対応する必要がある。
```

ここで大事なのは、変化への対応が簡単になるほど、判断軸が重要になること。

AI はたくさんの案を出せる。

でも、どの変化を受け入れるのか、どの変化を後回しにするのか、何を変えないのかは、人間が判断する必要がある。

特に、カスタマーの関心軸、事業上の優先順位、リスク、運用コストを見ずに変更を重ねると、動くけれどまとまりのないものになりやすい。

つまり、生成AI時代のこの価値は、こう変わる。

```text
計画に従うだけより、変化への対応
→
生成AIが変える社会や業務や顧客期待に、
計画を固定せず対応しながら、
何を変え、何を変えないかを構造化して判断する
```

言い換えるなら、こう。

```text
生成AIによって変わる世界に、計画を固定せず対応する。
```

または、少し記事向けにするなら、

```text
生成AIが変えていく現実に、アジャイルに対応する。
```

うぅ…AI で変えやすくなるほど、そして社会も変わりやすくなるほど、変える判断と変えない判断の両方が大事になる。

### 4つの価値のまとめ

|---|---|---|
| 人と対話 | 人と人が対話し、専門家が後で形にする | 人と人の対話に生成AIも加わり、その場で理解、構造化、試作を支える |

```text
4つの価値は残る。
でも、生成AIによって、

```

ここで大事なのは、アジャイル宣言が古くなるわけではないこと。

むしろ、生成AI時代に読み直すと、アジャイル宣言がなぜ今も効くのかが見えやすくなる。

具体的な手順ではなく、価値の置き方を書いているから、技術環境が変わっても読み直せる。

```text
アジャイル宣言は、生成AI時代に古くなるのではない。
```

うぅ…これは、おわりにの中心文にできそう。

## 「設計書よりも動くもの」も生成AI前後で変わる

アジャイル宣言の **Working software over comprehensive documentation** は、生成AI以前の文脈では、かなり重要な転換だった。

先に大きな設計書や要件書を固めるより、動くソフトウェアを作り、それを見ながら確認する。

これは、設計書が不要という意味ではない。

ただ、文書だけではわからないことがあり、動くものを通じて初めて見える価値や違和感がある、ということだった。

でも生成AI時代には、この言葉の重みが少し変わる。

生成AIによって、動くものを作るコストが大きく下がるから。

これまでは「動くものを作る」こと自体が、専門家の大きな仕事だった。

でも、2026年時点では、カスタマー自身や専門家が生成AIを使って、画面のたたき台や小さな業務ツールをかなり早く作れる。

つまり、生成AI以前のアジャイルでは、

```text
設計書だけでなく、動くものを作って確認しよう
```

だった。

ただし、その「動くもの」は、多くの場合、後で出てくるものだった。

打ち合わせで話し、専門家が持ち帰り、実装し、次のレビューで動くものを見る。

つまり、生成AI以前の `Working software` には、ある程度の時間差があった。

生成AI時代には、さらに進んで、

```text
後で動くものから、その場で動くものへ。
動くものは早く作れる。
では、それは本当に価値があり、安全で、運用できるものなのかをどう確かめるか
```

が重要になる。

「動くもの」の価値は下がらない。

でも、「動くものがある」だけでは、以前ほど強い証拠にならない。

生成AIでそれらしく動くものがすぐ出てくるからこそ、専門家は次を見る必要がある。

- その動きは、カスタマーの本当の関心軸に合っているか
- 例外や暗黙ルールを落としていないか
- データや権限の扱いは安全か
- 多人数で使ったときに破綻しないか
- 運用コストを読めるか
- 保守できる形になっているか

あの…生成AI時代の `Working software` は、「動いているもの」から「検証され、意味づけられた動くもの」へ重心が移るのかもしれない。

もう少し短く言うなら、こう。

```text
生成AI以前:
後で動くものを見て、対話する。

生成AI時代:
今すぐ目の前で動くソフトウェアを作りながら、対話する。
```

この変化はかなり大きい。

アジャイルが大事にしてきた「動くものを見ながら対話する」が、レビューの場だけでなく、対話そのものの中に入ってくるから。

## 対話のループに生成AIが入る

これまでのアジャイルでは、対話の中心には人がいた。

利用者や顧客が困りごとを話し、開発者や専門家が聞き、画面案や試作品にして、また見てもらう。

このループは、だいたい次のように見える。

```text
利用者のイメージ
→ 専門家との対話
→ 試作品
→ 利用者のフィードバック
→ 修正
```

生成AIが入ると、このループの中に、もう一つの作業主体が入ってくる。

```text
利用者のイメージ
→ 生成AIとの対話
→ 試作品
→ 利用者と専門家の確認
→ 修正
```

または、利用者が忙しい場合は、こうなる。

```text
利用者のイメージ
→ 専門家との対話
→ 専門家が生成AIで高速試作
→ 利用者と専門家の確認
→ 修正
```

つまり、生成AIは単なる実装補助ではなく、アジャイルが大事にしてきた **対話と試作のループそのもの** に入ってくる。

ここで大事なのは、生成AIが人間の対話を置き換えるとは限らないこと。

むしろ、対話の途中で試作品が出るまでの時間を短くする。

あの…「話す」と「作る」の間にあった距離が、かなり短くなる感じ。

その結果、専門家がいったん持ち帰って作業する時間は減っていくかもしれない。

これまでなら、打ち合わせで話を聞き、専門家が持ち帰り、画面案や試作品を作り、次の打ち合わせで見せる、という流れになりやすかった。

でも生成AIが対話の場に入ると、その場で画面のたたき台を出し、その場で違和感を直し、その場で次の案を試せる場面が増える。

```text
持ち帰って作る
→ その場で試す
→ その場で直す
→ その場で理解が深まる
```

もちろん、本番品質にする作業は持ち帰りになることも多い。

ただ、利用者のイメージを膨らませ、具体化し、試作品にして確認するところまでは、対話の場でかなり進むようになる。

うぅ…アジャイルの協調は、会議と実装の往復から、会話しながら作る形へ近づくのかもしれない。

## 専門家は Agent Skills や MCP で利用者を加速する

もう一つの変化は、専門家が直接アプリを作るだけではなく、利用者やチームが生成AIを使いやすくするための土台を用意すること。

たとえば、専門家は次のようなものを提供できる。

- 業務ごとの Agent Skills
- 安全なプロンプトや作業手順
- 参照してよい社内文書や仕様の整理
- MCP サーバーによる社内システムやデータへの接続
- 試作品生成用のテンプレート
- セキュリティや権限の制約を組み込んだ実行環境
- レビュー、テスト、デプロイの自動チェック

これによって、利用者は白紙の生成AIに丸投げするのではなく、専門家が整えた道具立ての中で、安全に試作品へ近づける。

ここでは、専門家の仕事は「毎回手で全部作ること」から、次のように変わる。

```text
利用者が速く、安全に試せる環境を作る。
AIが参照してよい情報と、してはいけないことを設計する。
生成されたものを本番に近づけるための検証ループを作る。
```

これは、生成AI時代のアジャイルにおける新しい専門家の役割としてかなり重要。

専門家は、アプリそのものだけでなく、**利用者がAIと対話して試作するための仕組み** を作る。

うぅ…そう考えると、Agent Skills や MCP は、単なる開発者向けツールではなく、利用者と専門家の協調を加速する部品として見えてくる。

## 情報を構造化する人間の役割はまだ大きい

生成AIが試作品を速く作れるようになっても、情報の構造化まで完全に任せられるかというと、2026年時点ではまだ人間がかなり役立つ。

利用者の話には、たいてい複数のものが混ざっている。

- いま困っていること
- 本当に達成したいこと
- 以前の運用の都合
- 例外的なケース
- 組織内の暗黙ルール
- 誰が責任を持つのか
- どこまで自動化してよいのか
- 何を作らないほうがよいのか

生成AIは、これらを受け取って要約や分類案を出すことはできる。

でも、ここで大事なのは、単にきれいに並べることではない。

お客様や利用者が何に関心を持っているのか。その関心ごとの軸を把握し、その軸に従って情報を構造化すること。

たとえば、同じ要望の集まりでも、軸の置き方で見え方が変わる。

- 業務フローの軸
- 利用者ロールの軸
- データの軸
- 権限と責任の軸
- リスクの軸
- コストの軸
- 頻度の軸
- 例外処理の軸
- 本番化の難しさの軸

どの軸で見るかによって、先に試作するもの、後回しにするもの、本番化の前に確認すべきものが変わる。

ここは、2026年時点ではまだ人間の判断が大きい。

特に、利用者、専門家、管理者、運用担当者の間で言葉の意味がずれているときは、人間が間に入って関心軸を見つける意味がある。

```text
生成AIは、情報の要約や分類を助ける。
でも、お客様の関心軸を把握し、
その軸に従って情報を構造化し、
どう合意に持っていくかは、
2026年時点ではまだ人間の仕事が大きい。
```

ここで専門家は、単に技術を知っている人ではなくなる。

利用者の言葉をほどき、関心軸を見つけ、その軸に沿って情報を構造化し、試作品に入れるものと入れないものを分け、合意できる形へ組み直す人になる。

あの…生成AIで試作が速くなるほど、その前後にある情報構造化の価値は、むしろ見えやすくなるのかもしれない。

ただし、この構造化も完全に手作業で残るとは限らない。

専門家が、関心軸の聞き取りや構造化を支援する Agent Skills を用意すれば、ここもかなり効率化できる。

たとえば、次のような Agent Skills が考えられる。

- 利用者の発話から困りごと、目的、制約、例外を分ける skill
- 業務フロー、利用者ロール、データ、権限、リスク、コストの軸で要望を分類する skill
- まだ確認できていない論点を質問として出す skill
- 試作品に入れるもの、後回しにするもの、本番化前に確認するものを分ける skill
- 利用者、管理者、運用担当者の関心差分を見える化する skill
- 合意事項、未決事項、前提条件を Markdown やチケットに落とす skill

この場合、人間の専門家は、毎回ゼロから聞き取り表を作るのではなくなる。

専門家は、構造化の型を Agent Skills として用意し、その出力を見て、関心軸がずれていないか、重要な論点が落ちていないかを確認する。

```text
人間が全部を手で構造化する
→ 専門家が構造化の型を Agent Skills として提供する
→ AI が下整理し、人間が関心軸と合意の妥当性を見る
```

うぅ…ここでも専門家の仕事は消えるというより、作業そのものから、作業を支える型の設計へ移っていく。

## 2026年時点の世間的な見え方

2026年時点では、生成AIを「専門家を加速する道具」とだけ見ると、少し足りない。

公開されている調査や研究でも、次のような傾向が見える。

- 生成AIは、定型的な実装、テスト、ドキュメント、ボイラープレート作成を短縮しやすい
- 一方で、要求分析や計画のような上流の曖昧な仕事では、効果が均一ではない
- 開発者の作業は、コード作成から、AI出力の指示、評価、修正へ移りつつある
- AI生成コードの品質、セキュリティ、検証負債が課題になっている

記事では、細かな数値よりも、次の整理を使いたい。

```text
生成AIは、まず専門家の作業を加速した。
次に、利用者自身が試作品へ到達する距離を縮めた。
そして今、本番品質を誰がどう担保するのかという問いを強くしている。
```

```text
生成AI時代のアジャイルでは、
動くソフトウェアが、開発者だけから出てくるとは限らない。
利用者とAIが作ったものも、対話の入口になる。
```

```text
専門家が持ち帰って作る時間は、少しずつ減っていく。
そのかわり、対話の場で試作品を作り、直し、確認する時間が増えていく。
```

## 注意して書きたいこと

- 「専門家不要論」にはしない
- 「生成AIはまだ使えない」という話にも寄せない
- 2025年のAI観だけで固定しない
- 2026年時点の変化の速さを入れる
- 未来断定は避ける
- 試作品と本番品質の違いを丁寧に分ける
