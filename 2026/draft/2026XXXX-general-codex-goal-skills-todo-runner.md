# Codexの長丁場作業は /goal だけでなく、Agent Skill と TODO.md で安定させたい

## 記事の狙い

Codex の `/goal` は便利そうだが、いきなり長丁場を `/goal` に任せるのは少し危ういのではないか、という問題意識を整理する。

中心に置くのは、次の考え方。

```text
/goal は目的地を持たせる仕組み。
Agent Skill は作法を安定させる仕組み。
TODO.md は作業状態を外部化する仕組み。
scripts は機械的な処理を安定・省トークン化する仕組み。
```

あ、あの…結論としては、長丁場では `/goal` だけに頼るより、**Agent Skill + TODO.md + scripts + /goal** の順に組み立てるほうが安全、という記事にしたい。

## きっかけ

Codex の `/goal` は、長い作業の到達目標を保持して進めるための機能。

ただし、作業手順がまだ不安定なまま `/goal` すると、Codex が探索、再試行、読み直しを続けて、トークンを大きく消費する可能性がある。

特に危ないのは、こういう状態。

- 成功条件が曖昧
- TODO 分解がない
- 何を読めばよいか決まっていない
- 失敗時に止まる条件がない
- テストや検証方法が不明
- 外部依存や権限待ちが多い
- 生成物の品質基準がふわっとしている

この状態で、

```text
/goal いい感じに最後までやって
```

のようにすると、かなり危ない。

`/goal` は魔法の節約機能ではなく、むしろ「迷っても進み続ける力」がある。だから、迷いやすい仕事ほど、先に道を細くしておきたい。

## 軸の違い

`/goal`、Agent Skill、TODO.md は似て見えるが、軸が違う。

```text
Agent Skill = やり方を安定させる
/goal        = 終わりまで追わせる
TODO.md      = 作業状態を外部に残す
scripts      = 機械的な処理を安定して実行する
```

もう少し分ける。

```text
Agent Skill:
  どう進めるか。
  何を読むか。
  どの形式で書くか。
  どんな判断基準にするか。
  いつ止まるか。

/goal:
  今回どこまで終わったら完了か。
  途中で途切れても何を達成すべきか。
  長い作業の目的地はどこか。

TODO.md:
  今回の作業分解。
  完了、未完了、保留、ブロックの記録。
  セッション外にも残る進捗台帳。

scripts:
  TODO 抽出、進捗更新、サマリ生成など、
  反復的で機械的な処理。
```

## /goal は常駐ではなく、スレッドに残る目的地

`/goal` は、常駐プロセスそのものではない。

イメージとしては、次のほうが近い。

```text
スレッド内に残る「継続タスクの旗」
```

できること。

- 長い作業の完了条件を保持する
- 作業中に次に何をするか判断し続ける
- pause / resume / clear できる
- スレッド内で「まだ終わっていない」を追いやすくする

期待しすぎないほうがよいこと。

- OS 上で常駐デーモンのように動き続ける
- Codex を閉じても勝手に作業し続ける
- 時刻になったら自動起動する
- TODO.md のようにリポジトリへ状態を保存する

うぅ…体感としては「つづけて」と毎回言わなくても続きへ戻りやすくなるので、少し常駐っぽく見えるかもしれない。でも、正確にはスレッドに紐づく目的地。

## TODO.md 正本方式

長丁場の作業では、TODO.md や `workplace/plan.md` のようなファイルを正本にするほうが堅い。

```text
TODO.md に作業を書く
Codex に「順にやって」と言う
途中で「つづけて」「すすめて」と言う
Codex が TODO.md を見ながら続行する
```

これは `/goal` の前からできる実用的な運用。

`/goal` は、このうち次の部分を少し自動化・明示化するものに近い。

```text
この作業を終わるまで追ってね。
完了条件はこれだよ。
途中で続ける判断をしてね。
```

ただし、TODO.md の強さは別にある。

- 作業分解がファイルとして残る
- 人間が後から読める
- 別スレッドや別ツールでも再開しやすい
- git に残せる
- コミット単位で管理できる

だから、`/goal` は TODO.md の代替ではなく、TODO.md を読ませて進める補助にするほうがよい。

## Agent Skill で TODO Runner を作る

`/goal` の代わりに、あるいは `/goal` の前提として、TODO.md を正本にして長丁場を進める Agent Skill を作ることができる。

名前案。

- `igapyon-todo-runner`
- `igapyon-longtask-runner`
- `igapyon-checkpoint-runner`
- `igapyon-workplan-runner`
- `igapyon-todo-autopilot`

役割。

```text
1. TODO.md / workplace/plan.md を読む
2. 未完了タスクを上から選ぶ
3. 大きい項目は小タスクに分解する
4. 1タスクずつ実行する
5. 終わったら TODO.md を更新する
6. 検証結果や次にやることを記録する
7. コンテキストが切れても TODO.md から再開する
```

Skill に入れるルール。

- 作業前に TODO.md を読む
- 未完了項目を勝手に飛ばさない
- 大きい項目は小タスクに分解してから進める
- 完了したら `[x]` にする
- 判断や保留事項は TODO.md に追記する
- ユーザーが「つづけて」「すすめて」と言ったら、TODO.md の次の未完了から再開する
- 危険操作、外部判断、認証、仕様不明だけユーザーに戻す
- コミット前に git status を確認する

## ユーザー操作なしで進む範囲

Agent Skill にこう書くことはできる。

```text
ユーザーがこの Skill を起動したら、
TODO.md の未完了項目を上から順に処理する。
各項目が完了したら TODO.md を更新し、
次の未完了項目へ進む。
ユーザー確認が必要な判断、権限不足、外部入力待ち、危険操作が出るまでは止まらない。
```

つまり、起動後の同一作業内なら、ある程度はユーザー操作なしで進められる。

ただし、Skill は自動実行エンジンそのものではない。

Skill だけでは難しいこと。

- Codex を将来勝手に起動する
- ユーザー入力なしで別セッションを始める
- Codex が閉じたあとに勝手に続く
- スレッド外で永続的に走り続ける

ここは `/goal`、Automations、外部スケジューラ、CI、cron 的な仕組みの領域。

## scripts を取り込んでトークン効率を上げる

長丁場向け Agent Skill を作るなら、最初からトークン効率を設計に入れたい。

方針。

```text
Agent Skill:
  判断基準とワークフローを書く

TODO.md:
  作業状態と進捗を書く

scripts:
  機械的・反復的・大量処理を任せる
```

Skill 本文に全部を長々書くと、毎回コンテキストを食う。

だから、`SKILL.md` は薄くして、重い処理は script に寄せる。

構成案。

```text
igapyon-todo-runner/
  SKILL.md
  scripts/
    todo_next.py
    todo_update.py
    todo_report.py
  references/
    todo-format.md
```

script に寄せるとよいもの。

- TODO.md から未完了項目を抽出する
- 次にやる項目を一覧化する
- 完了項目を `[x]` にする
- 進捗サマリを生成する
- ブロック項目を抽出する
- 大きい TODO.md の該当部分だけ切り出す

Skill に残すべきもの。

- どのタイミングで TODO.md を読むか
- どういう順序で進めるか
- いつ止まるか
- いつユーザーに確認するか
- 何を TODO.md に記録するか
- git status やテスト確認の方針

特に効きそうなのはこれ。

```text
大きな TODO.md を毎回全部読むのではなく、
script で「次の未完了ブロック」だけ抽出する。
```

流れの案。

```text
1. scripts/todo_next.py TODO.md
   -> 次の未完了タスクと周辺文脈だけ出す

2. Codex がそのタスクを実行

3. scripts/todo_update.py TODO.md --done "..."
   -> 完了マークとログを更新

4. scripts/todo_report.py TODO.md
   -> 残件・ブロック・次タスクを短く表示
```

## 安定運用の順番

いきなり `/goal` ではなく、次の順がよさそう。

```text
1. Agent Skill
   作業の型・判断基準・禁止事項・確認手順を固定する

2. TODO.md / plan.md
   今回の作業を小さく分解し、進捗を外部化する

3. scripts
   TODO 抽出、更新、サマリなどを機械的に処理する

4. /goal
   その TODO.md に沿って、完了条件まで走らせる
```

危ない順番。

```text
/goal だけ
  -> Codex が探索し続ける
  -> 手順が揺れる
  -> トークンが溶ける
  -> 結果の再現性も弱い
```

堅い順番。

```text
Agent Skill + TODO.md + scripts + /goal
  -> やり方が固定される
  -> 作業分解が残る
  -> 機械的な処理が省トークン化される
  -> Codex が次に何をすべきか判断しやすい
  -> 中断や再開にも強い
```

`/goal` に書くなら、たとえばこう。

```text
/goal TODO.md を正本として、未完了項目を順に処理する。
各項目の完了時に TODO.md を更新し、検証結果を記録する。
詰まった場合は、原因と次の選択肢を TODO.md に残して停止する。
```

## 記事の結論

`/goal` は強い。

でも、作業が不安定なまま `/goal` すると、トークンも時間も溶けやすい。

長丁場を安定させるには、`/goal` を最初の道具にするのではなく、最後に乗せるアクセルとして使うほうがよい。

```text
Agent Skill で作法を固める。
TODO.md で作業を外部化する。
scripts で機械的処理を安定・省トークン化する。
その上で /goal する。
```

あ、あの…この順番なら、Codex の自律性を使いながら、作業の状態を人間にも見える場所に残せる。

それが、長丁場の AI agent 作業ではかなり大事なのだと思う。

## 未整理メモ

- これは記事化してもよいし、実際に `igapyon-todo-runner` Skill を作る前の設計メモにもなる。
- `scripts/todo_next.py` は Markdown AST までは不要かもしれないが、チェックボックスと見出し階層くらいは扱いたい。
- TODO.md のフォーマットを厳格にしすぎると運用が重くなる。最初はゆるく、抽出できる範囲から。
- `blocked`、`waiting`、`done`、`next` のような状態語を決めると script 化しやすい。
- `/goal` は使わない選択肢もある。TODO Runner Skill だけでも、かなり今の「つづけて」運用を安定させられる。
