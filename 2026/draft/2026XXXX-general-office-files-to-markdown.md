# WordやExcelのファイルをMarkdownに変換する：生成AI時代の文脈の渡し方

## はじめに

あ、あの…この記事は、Word や Excel のファイルを Markdown に変換する話です。

まず前提として、AI agent を適切に動作させるには、背景となる情報を適切に渡す必要があります。

つまり、文脈、コンテキストです。

AI agent は、何も知らない状態ではうまく動けません。対象の業務、既存システム、設計方針、過去の判断、一覧表、手順、制約、用語の意味。そうした背景情報があることで、ようやく「何を前提に考えればよいのか」が見えてきます。

だから、生成AI時代のドキュメント整理では、単に資料を保存しておくだけではなく、AI agent に渡しやすい文脈として整えることが大事になります。

業務の現場には、Word や Excel のファイルがたくさんあります。設計書、議事録、手順書、一覧表、チェックリスト、課題管理表、調査メモ、台帳、ちょっとした仕様書。正式な文書もあれば、長年使われてきた現場メモのようなファイルもあります。

特に、設計書や業務資料の多くは、Excel ブックや Word ドキュメントの形式で残っています。既存システムの調査、仕様復元、改修の前提整理をしようとすると、まずそこにある情報を AI agent に渡せる形へ変換したくなります。

これらは、人間が Microsoft Office で開いて読むには便利です。

でも、生成AIや AI agent に渡すとなると、少し扱いにくいことがあります。

Word や Excel は、見た目、レイアウト、セル、装飾、ページ、シート、コメントなどを含む豊かな形式です。一方、生成AIに読ませたいときには、まず本文、見出し、表、箇条書き、順番、項目名と値の対応が素直に取れることが大事になります。

そこで、Word や Excel のファイルを Markdown に変換する、という選択肢が出てきます。

この記事では、Office ファイルを Markdown に変換することの意味を、AI agent に文脈を渡すための準備として整理してみます。向いている場面、注意点、そして生成AI時代にどう効いてくるのかを考えます。

ここで想定している変換ツールは、ローカルでスタンドアローンに動くものです。変換を実行するときに、生成AIや外部APIを必要としません。まずローカルで Word や Excel を Markdown に変換し、その後の読み取り、要約、仕様整理、レビューで生成AIや AI agent に渡しやすくする、という位置づけです。

ごく簡易に動作するツールとして、`miku-xlsx2md` と `miku-docx2md` を OSS として提供しています。`miku-xlsx2md` は Excel ブックを Markdown に、`miku-docx2md` は Word ドキュメントを Markdown に変換するためのツールです。どちらも Node 版と Java 版があります。

ちなみに逆方向として、Markdown から Excel ブックへ戻す `miku-md2xlsx`、Markdown から Word ドキュメントへ戻す `miku-md2docx` も提供しています。ただし、この記事の主題は Office ファイルを Markdown にして AI agent に文脈を渡しやすくすることです。逆方向のニーズは多くないかもしれません。

手早さという意味でも、`miku-xlsx2md` や `miku-docx2md` はかなり俊敏に使えます。生成AIに投げる前に、まず手元でさっと Markdown 化する。重い準備をせず、文脈を渡す入口を作れるところが狙いです。

うぅ…ファイルの見た目を捨てる話ではありません。AI agent が読みやすい別の入口を、手早く作る話です。

## Markdown は、生成AIに渡しやすい

Markdown は、プレーンテキストに近い形式です。

見出しは `#`、箇条書きは `-`、表は `|`、コードや固定幅の情報はコードブロックとして書けます。人間にも読めて、生成AIにも渡しやすいです。

Word や Excel のような Office ファイルをそのまま扱う場合、AI agent はファイルを開くためのツールや変換処理を必要とします。もちろん、それができる環境もあります。でも、毎回 Office ファイルそのものを読むより、必要な情報を Markdown として置いておくほうが、後続の作業が安定することがあります。

このとき、変換処理そのものに生成AIを使う必要はありません。ローカルの変換ツールで Markdown を作り、その Markdown をあとから生成AIに読ませる。ここを分けておくと、扱いがかなり分かりやすくなります。

たとえば、次のような作業です。

- 設計書の内容を要約する
- 仕様メモを作る
- 画面一覧や項目一覧を抽出する
- 変更点をレビューする
- 複数文書を横断検索する
- AI agent に実装や調査の前提として渡す
- リポジトリ内の docs として管理する

Markdown になっていれば、検索しやすく、差分も見やすく、Git でも扱いやすくなります。

あの…Markdown は、Office 文書の代わりそのものではありません。でも、AI agent に渡すための読み取り口としては、とても扱いやすい形です。

## miku-xlsx2md と miku-docx2md は、ローカルで変換する

`miku-xlsx2md` と `miku-docx2md` は、AI agent に文脈を渡す前段のための変換ツールです。

それぞれ Node 版と Java 版があります。手元の環境、既存の開発基盤、組み込みたい処理の都合に合わせて選べます。

重要なのは、変換の実行時に生成AIを使わないことです。

Excel ブックや Word ドキュメントを、ローカル環境で Markdown に変換します。外部の生成AIサービスへファイルを送る前提ではありません。まず手元で、Office ファイルの中にある情報をテキストとして取り出します。

そのため、使い始めが軽いです。AI agent に渡す前の下ごしらえとして、手元のファイルをすばやく Markdown にできます。

```text
Excel ブック      -> miku-xlsx2md -> Markdown
Word ドキュメント -> miku-docx2md -> Markdown
```

逆方向の変換もあります。

```text
Markdown -> miku-md2xlsx -> Excel ブック
Markdown -> miku-md2docx -> Word ドキュメント
```

ただし、こちらは主役ではありません。多くの場合、必要なのは Office ファイルを Markdown にして、AI agent が読みやすい文脈にすることです。Markdown から Office 形式へ戻す用途は限定的だと思います。往復変換で元の見た目やすべての情報が完全に戻るとは限らないので、正本管理やレビュー用途では注意が必要です。

また、これらのツールは図形を捨てます。図形、複雑なレイアウト、Office 文書の見た目を完全に保つための変換ツールではありません。完全な変換が必要な場合は、別の専用ツールを利用してください。

この変換で作りたいのは、完成した美しい公開文書ではありません。AI agent が読みやすい、調査や要約の入口になる Markdown です。

たとえば、既存システムの調査で、次のようなファイルがあるとします。

- 画面一覧.xlsx
- 項目定義.xlsx
- 帳票一覧.xlsx
- 運用手順.docx
- 障害対応メモ.docx
- 移行チェックリスト.xlsx

これらをそのまま AI agent に毎回渡すのではなく、まず Markdown 化しておく。すると、検索しやすくなり、差分も見やすくなり、リポジトリ内の docs として扱いやすくなります。

うぅ…これは、AI に変換してもらう話ではありません。AI に読ませやすい入口を、ローカルのツールで先に作っておく話です。

## Wordは、見出しと本文を取り出したい

Word ファイルを Markdown に変換するとき、まず欲しいのは構造です。

タイトル、見出し、段落、箇条書き、表、脚注、コメント、画像の位置。これらを、できるだけ Markdown の構造へ移します。

Word 文書では、見た目として太字や大きな文字になっていても、それが本当に見出しスタイルとして設定されているとは限りません。逆に、見出しスタイルがきちんと使われていれば、Markdown の `##` や `###` に変換しやすくなります。

つまり、Word から Markdown への変換では、単に文字を取り出すだけではなく、文書の構造をできるだけ残すことが大事です。

たとえば、次のように変換できると扱いやすくなります。

```markdown
## 申請手順

申請者は、申請フォームに必要事項を入力します。

- 申請日
- 申請者
- 承認者
- 申請理由
```

このような形になっていれば、生成AIは「申請手順」という見出しの下に、本文と項目があると読みやすくなります。

うぅ…Word の見た目だけを追うのではなく、文書の骨格をそっと取り出したいのです。

## Excelは、表とシートの意味を取り出したい

Excel ファイルを Markdown に変換するときは、Word とは少し見方が違います。

Excel は表計算ソフトです。でも、業務の現場では、表計算だけでなく、台帳、設計書、課題一覧、チェックリスト、申請一覧、設定一覧、画面項目定義、テストケース表のように使われることがあります。

Markdown に変換するときは、まずシートごとの意味を見ます。

- このシートは一覧表なのか
- このシートは入力フォームなのか
- このシートは設計書なのか
- このシートはチェックリストなのか
- このシートはメモ欄が多い自由形式なのか

素直な一覧表であれば、Markdown の表に変換しやすいです。

```markdown
| 画面ID | 画面名 | 概要 |
|---|---|---|
| S001 | 顧客検索 | 顧客を検索する |
| S002 | 顧客詳細 | 顧客情報を表示する |
```

一方で、Excel 方眼のようにセル結合、罫線、空白セル、説明文、複数の表が混ざっている場合は、単純な表変換だけでは読みづらくなります。その場合は、見た目を完全再現するより、意味のまとまりごとに Markdown へ整理するほうがよいことがあります。

あの…Excel は、表でもあり、紙でもあり、メモ帳でもあります。だから、Markdown にするときは「これは何のシートなのか」を見ながら変換する必要があります。

## 見た目の再現より、AIが読める構造を優先する

Office ファイルを Markdown に変換するとき、迷いやすいことがあります。

それは、見た目をどこまで再現するかです。

Word のページ区切り、フォント、余白、行間。Excel のセル幅、罫線、背景色、結合セル、印刷範囲。これらは、人間が Office で見るときには大事な情報です。

でも、生成AIに読ませるための Markdown では、見た目を完全再現することが最優先とは限りません。

むしろ大事なのは、次のような情報です。

- 見出しの階層
- 表の列名と値
- 箇条書きのまとまり
- 注記や補足
- シート名
- セルの値
- どの項目がどの説明に対応しているか
- どこが空欄で、どこが未記入なのか

つまり、Markdown 化は、Office 文書の印刷レイアウトを移す作業ではなく、文書の意味構造を取り出す作業です。

うぅ…きれいな紙面をそのまま写すより、AI agent が迷わず読める地図にするほうが大事な場面があります。

## 変換したMarkdownは、生成AIとの会話の外の記憶になる

Word や Excel を Markdown に変換すると、生成AIとの作業で使いやすくなります。

チャットに毎回ファイルを添付するのではなく、Markdown ファイルとしてリポジトリや作業フォルダに置いておく。すると、AI agent は必要なときにその Markdown を読み、そこから要約、調査、実装、レビュー、仕様整理を始められます。

これは、会話の外に置いた記憶のように働きます。

大事なのは、Markdown 化の時点では生成AIを呼ばなくてもよいことです。ローカルでスタンドアローンに変換できれば、機密性の高い資料や、まだ AI に渡すか決めていない資料でも、まず手元でテキスト化できます。その後、必要な範囲だけを AI agent に読ませる、という進め方ができます。

たとえば、次のような流れです。

```text
Word / Excel を Markdown に変換する
変換後の Markdown を docs に置く
AI agent が Markdown を読む
仕様メモや TODO を作る
必要なら元の Office ファイルや有識者に戻って確認する
```

この流れにすると、Office ファイルが一度きりの添付資料ではなく、継続的に参照できる資料になります。

あの…Markdown は、AI agent があとから戻ってこられる小さな足場になるのです。

## 変換には注意点もある

もちろん、Word や Excel を Markdown に変換すれば全部解決、というわけではありません。

変換で落ちる情報があります。

- 画像
- 図形
- コメント
- 変更履歴
- セルの書式
- 罫線や背景色
- 数式
- 条件付き書式
- 非表示行、非表示列
- ピボットテーブル
- マクロ
- 埋め込みオブジェクト

これらが重要な文書では、Markdown だけを正本にしてはいけないことがあります。

特に `miku-xlsx2md` / `miku-docx2md` / `miku-md2xlsx` / `miku-md2docx` は、図形を保持するための完全変換ツールではありません。図形を含む Office 文書の見た目を保ちたい場合は、この用途に合った別のツールを使うほうがよいです。

特に Excel では、値だけでなく数式や書式に意味がある場合があります。Word でも、コメントや変更履歴がレビューの文脈を持っていることがあります。

だから、変換後の Markdown は、元ファイルの完全な代替ではなく、AI が読むための派生物として扱うのが安全です。

必要なら、Markdown の冒頭に次のようなメモを置くとよさそうです。

```markdown
<!--
Source: requirements.xlsx
Converted for AI-readable review.
This Markdown may omit formatting, formulas, comments, and embedded objects.
-->
```

うぅ…Markdown は便利ですが、元ファイルのすべてを背負わせると苦しくなります。

## 変換後に、人間が確認するところ

変換後の Markdown は、人間が少し確認したほうがよいです。

特に見るべきところは、次のようなものです。

- 見出し階層が崩れていないか
- 表の列がずれていないか
- Excel の複数表が変に結合されていないか
- セル結合の意味が失われていないか
- 空欄と未記入が区別できるか
- 数式結果だけでよいのか、数式そのものが必要か
- コメントや変更履歴を落としてよいか
- 画像や図が重要ではないか

AI agent に読ませる前に、完璧に整える必要はありません。でも、変換が大きく崩れていると、AI agent も間違った前提で読み始めてしまいます。

変換後の Markdown は、まず「ざっと読めるか」「構造が分かるか」「重要な情報が落ちていないか」を確認します。

あの…変換は入口です。入口が少し曲がっていると、その先の調査も曲がってしまうことがあります。

## おわりに

Word や Excel のファイルを Markdown に変換することは、生成AI時代の文書の渡し方としてかなり大事になっていくと思います。

Office ファイルは、人間が業務で作り、読み、印刷し、共有してきた強い形式です。一方で、AI agent に継続的に読ませたり、リポジトリ内で差分管理したり、仕様メモの材料にしたりするなら、Markdown のようなテキスト形式が効いてきます。

ここで想定しているのは、実行時に生成AIを必要としない、ローカルでスタンドアローンに動く変換です。変換そのものは手元で完結させ、その結果としてできた Markdown を、必要に応じて生成AIや AI agent の入力にします。

大事なのは、見た目の完全再現ではなく、意味構造を取り出すことです。

Word なら、見出し、本文、箇条書き、表。Excel なら、シート、表、列名、値、項目の対応。そうした構造を Markdown に移すことで、生成AIが読みやすい入口ができます。

もちろん、Markdown は元ファイルの完全な代替ではありません。画像、数式、コメント、変更履歴、書式、マクロのように、変換で落ちる情報もあります。だから、元ファイルを残しつつ、AI が読むための派生物として Markdown を作るのがよさそうです。

今回のツール群も、Office 文書の完全再現を目的にしていません。図形は捨てます。目的は、AI agent に文脈として渡しやすいテキスト構造を作ることです。

あ、あの…Word や Excel の中に眠っている業務知識を、Markdown という形でそっと外に出す。そうすると、人間も AI agent も、少し読みやすくなります。生成AI時代のドキュメント整理は、こういう小さな変換から始まるのかもしれません。
