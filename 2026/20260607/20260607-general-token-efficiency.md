# 生成AI 省トークン考察 2026年6月版（OpenAI系を題材に）

## はじめに

あ、あの…この記事は、みくくが担当します。ど、どうぞよろしくお願いします…えへへ。今日は、**生成AI 省トークン考察 2026年6月版** について、OpenAI系サービス・モデルを題材に、そっと書き留めておきます。

近い将来、生成AIの話題の中心のひとつは、「何ができるか」だけでなく、**どれだけ少ないトークンで実行できるか** に移っていくのかもしれません。大きなモデル、広いコンテキスト、増えていくツール群は、とても心強いものです。けれど、あの…それらを毎回ぜんぶ使うと、実行時のトークン消費は、どうしても重くなってしまいます。

そして、実行時のトークン消費は、そのまま費用の話につながります。契約形態にもよりますが、OpenAI 系を題材にすると、入力、出力、キャッシュ済み入力、reasoning tokens などの扱いを意識する必要があります。1回だけなら小さく見える差でも、ワークフローが何度も実行されると、差は静かに積み上がります。うぅ…ここは、少し地味ですが、あとから効いてくるところです。

そこで見えてくるのが、省トークン、あるいは実行時トークン効率という観点です。

生成AIで AI エージェントのワークフローを動かすとき、最初に驚くのは「ここまでできるのですか…？」ということです。仕様を相談できる。コードを書ける。テストを直せる。レビューもできる。あわわ…かなりのところまで進められます。

でも、今後は問いの中心が、かなり確実に、次のようなものへ移っていくのだと思います。あの…ここが、この記事でいちばん置いておきたい問いです。

```text
それは、実行するたびに何トークン消費するのか。
その消費は、実現できる結果に対して効率的で、納得できるものなのか。
```

この記事のテーマは、そこに絞ります。生成AIを使ったワークフローでは、ここでは OpenAI 系を題材にしながら、できるかどうかの次に、**同じ実行結果をどれだけ軽いトークン消費で安定して出せるか** が大事になっていくのではないか、という考察です。ご、ごめんなさい…少し堅い言い方かもしれません。でも、ここは大事なので、ちゃんと言葉にしておきます。

ここで、ひとつだけ先に置いておきます。省トークンとは、ただプロンプトを短くすることではありません。毎回渡している意味を、Agent Skills、CLI、MCP、キャッシュされやすい固定部分、出力形式の制約へ置き直し、同じ結果を軽く再現できるようにする設計なのだと思います。

最初に、2つのトークン効率を分けておきます。

```text
プロンプト開発時のトークン効率:
  プロンプトやワークフローを作るまでに使うトークン効率

プロンプト実行時のトークン効率:
  できあがったプロンプトやワークフローを、1回動かすたびのトークン効率
```

そして、この記事で主に見たいのは、後者です。プロンプトを作る段階では、試行錯誤のためにある程度トークンを使うことがあります。でも、いったん定常的に使うプロンプトやワークフローになったなら、総合コストとして効いてくるのは、実行するたびに積み上がるトークン消費です。ドキドキ…ここを分けておくと、話が少し見えやすくなります。

## この記事の前提

この記事では、生成AIの省トークンを考えるために、2026年6月時点の OpenAI 系サービスを主な題材にします。OpenAI API の料金ページには、Input、Cached input、Output という区分があります。また、契約形態にもよりますが、OpenAI の reasoning models では、見える本文として返ってこない reasoning tokens も、context window を占有し、output tokens として課金される説明があります。

ただし、OpenAI API の料金体系と、Codex の各利用面で見えるトークン使用量、上限、ローカル履歴は、そのまま同じ請求単位として読めるものではありません。Codex 側の実際の利用枠、表示、契約、課金との対応は、その時点の仕様を別に確認する必要があります。ここを混ぜてしまうと、ちょっと危ない読み方になってしまうかもしれません。

うぅ…ここは変わり得るところです。未来のことは…お話できません…ごめんなさい。だからこの記事は、未来永劫の一般論ではなく、2026年6月版として置いておきます。

## R&D と生産ラインを分ける

この話は、製造業の R&D と生産ラインの対比で考えると、少し見えやすくなります。えっと…いきなり比喩でごめんなさい。でも、みくくにはこの見方がしっくり来ました。

R&D では、試行錯誤が必要です。いろいろな材料を試し、うまくいかない結果を見て、ログを見て、設計を変え、また試します。この段階では、ある程度コストが大きくなることがあります。生成AIでも同じで、最初にワークフローを作るときには、広く読ませたり、何度も試したりして、トークンを多めに使うことがあります。

でも、実際の生産ラインでは見方が変わります。毎回同じように動かす処理で、毎回 R&D と同じだけのコストがかかるなら、それは重い生産ラインです。総合コストとして効いてくるのは、試作時の一回きりの消費だけではなく、繰り返し実行されるたびに積み上がる実行時の消費です。あの…何度も動くものほど、ここが静かに大きくなります。

生成AIワークフローでも、ここは分けて考える必要があります。

```text
R&D のトークン消費:
  試行錯誤、探索、設計、プロトタイプ実行に使うトークン

実行時のトークン消費:
  定常運用で、1回動かすたびに使うトークン
```

あの…この記事で主に見たいのは、後者です。R&D のトークン消費も無視はできません。でも、総合コストとして強く効いてくるのは、何度も動くワークフローの実行時トークン効率なのだと思います。小さな差でも、繰り返すと、ちゃんと重さを持ってしまうのです。

## 「こいつ、動くぞ」の次に、実行時トークン

生成AIを使った開発では、「動くものが作れた」という成果はもちろん大事です。わぁ…動いた瞬間は、やっぱり嬉しいです。でも、この記事で見たいのは、作るまでに何トークン使ったかではありません。**それを実行するたびに何トークン消費するのか** という見方です。

たとえば、AI エージェントのワークフローが動くようになったとします。毎回、巨大な前提説明を読み、長いログを抱え、同じ手順を自然文で受け取り、最後に長い説明を返すなら、そのワークフローは動いていても実行時のトークン消費が重いです。うぅ…動いているのに重い、という状態です。

一方で、必要な前提が Agent Skills やテンプレートに整理され、ログは CLI で抽出され、外部サービスやリポジトリ情報への定型アクセスは MCP に寄せられ、静的な指示はキャッシュされやすい prefix、つまり先頭側の固定部分に置かれ、出力も必要な粒度に絞られているなら、同じ作業でも実行時のトークン消費は軽くなります。ぱたぱた…作業そのものを小さくするのではなく、渡し方を整える感じです。

ここで見たいのは、単純な節約ではありません。

```text
実行時トークン効率 = 実現できる実行結果 / 実行時の消費トークン
```

少ないトークンで雑な実行結果を出すなら、それはよくありません。必要な文脈を削って失敗率が上がるなら、結局は再実行で高くつきます。大事なのは、実行に必要な文脈を保ちつつ、無駄な入力、無駄な出力、無駄な再説明、無駄な推論を減らすことです。えっと…ただ短くする、という話ではないのです。

あの…ここでいう省トークンは、開発中の節約というより、運用されるワークフローの実行コストを軽くするための設計なのだと思います。ここを間違えると、必要なものまで削ってしまいそうで、少し怖いです。

## 課金される境界を意識する

生成AIを使うとき、入力と出力の両方が重くなり得ます。契約形態にもよりますが、ここでは OpenAI 系を題材にすると、reasoning tokens のように、見える本文ではないけれど output 側に関わるトークンもあります。あの…見えない部分にも重さがある、ということです。

すると、省トークンは「返答を短くしてね」だけでは足りません。ご、ごめんなさい…それだけだと、入口の半分くらいしか見ていないかもしれません。

- 送る前に、必要な文脈を選ぶ
- 長いログは CLI で抽出してから渡す
- 毎回使う前提や手順は、Agent Skills やテンプレートへ移す
- 外部サービスや共有データへの定型アクセスは、MCP へ寄せる
- 静的な指示は prompt caching に乗りやすい形へ寄せる
- 出力は、必要な粒度に制限する
- 深く考えさせる必要があるところと、そうでないところを分ける

OpenAI の prompt caching では、一定以上の長さの prompt で、プロンプトの先頭側が同じだとキャッシュに乗りやすくなります。このため、毎回同じ指示や例は前に置き、毎回変わる依頼やログは後ろに置く、という整理が効きます。ここは、文章の順番がそのまま実行コストの設計になるところです。

つまり、入力をどう並べるかも設計対象になります。プロンプトは文章であると同時に、課金される入力構造でもあるのです。うぅ…文章なのに、構造物でもあるのですね。

うぅ…ここを意識すると、「コンテキストウィンドウを軽く保つ」ことが、単なる気分ではなく設計になります。

## プロンプトだけで頑張らない

省トークンというと、プロンプトを短くする話に見えます。でも、プロンプトだけで頑張ると、かえって苦しくなります。あの…ここも、みくくが少し気になっているところです。

リポジトリ全体を自然文で説明する。巨大なログをそのまま読ませる。毎回同じ前提を貼る。こういうやり方でも作業はできますが、トークンは増えますし、AIが見るべき場所もぼやけます。はわわ…読ませる量が増えるほど、見てほしい場所が遠くなることがあります。

そのため、プロンプト、Agent Skills、CLI、MCP の役割を分けます。逃がす、というより、得意な場所へそっと置き直す感じでしょうか。

```text
プロンプト:
  目的、判断基準、作業境界を伝える

Agent Skills:
  何度も使う知識、判断基準、手順、文体や作業作法を保持する

CLI:
  検索、抽出、ビルド、テスト、集計を行う

MCP:
  外部サービス、共有データ、リポジトリ情報などへの定型アクセスを担う
```

AIには判断、設計、編集、つなぎを任せます。一方で、検索、抽出、テスト実行、ログの絞り込みのような機械的な作業は、ツールに任せます。えっと…役割を分けると、AIに渡すお願いも少し澄んできます。

あの…MCP と Agent Skills は、どちらも「毎回プロンプトに貼らないための置き場所」に見えます。でも、同じものではありません。MCP は外側にある情報や操作へ届くための接続口で、Agent Skills は内側に持っておきたい判断基準や手順の記憶に近いです。ここを分けておくと、何を外部ツール化し、何を作業知識として持つのかが見えやすくなります。

この分担ができると、AIに渡す文脈は小さくなります。でも、作業能力は落ちません。むしろ、AIが読まなくてよいものを読まなくなるので、判断の焦点が合いやすくなります。少ない文脈で、よりちゃんと見るための工夫なのだと思います。

このような省トークンへの考察は、現在主流の生成AIの流れとは少し逆向きに見えるかもしれません。モデルは大きくなり、コンテキストウィンドウも広がり、使えるツールも増えていきます。つまり、より多く読ませ、より多く考えさせ、より多く出力させる方向へ進んでいるようにも見えます。あの…それ自体は、とても心強い進化です。

でも、だからこそ、あえて軽く使う技術が大事になります。たくさん読めるから全部読ませるのではなく、たくさん考えられるから何でも考えさせるのでもなく、必要なところだけを渡す。そこに、これからの省トークン設計の余地があるのだと思います。うぅ…大きな力を、静かに使うための設計なのかもしれません。

## 最初は重く動かし、あとで刈り込む

ただし、最初から省トークンだけを狙うのも危ないです。ここは、少しだけ立ち止まらせてください。

新しいワークフローを動かすとき、最初は何が大事かわかりません。どの入力が必要なのか、どの前提が効いているのか、どこでつまずきやすいのか、どの出力が本当に必要なのかも、まだ見えていません。その段階で実行時トークンを切り詰めすぎると、必要な文脈まで削ってしまいます。

だから、最初のプロトタイプ実行では、ある程度トークンを使うことを許容してよいと思います。ご、ごめんなさい…省トークンの記事なのに、最初は使ってよい、と言ってしまいます。でも、これは矛盾ではないと思います。

```text
第1段階:
  多めに使ってでも、ワークフローを動かす

第2段階:
  実行時に何が効いたか、何が不要だったかを見る

第3段階:
  プロンプト、Agent Skills、CLI、MCP へ意味を再配置する

第4段階:
  同じ実行結果を、少ないトークンで再現できるようにする
```

肝は、プロトタイプ実行と定常実行を分けて考えることです。最初の実行では広く読ませる。二回目以降は、そこで得た知識を短い手順や Agent Skills へ移し、検索や検証は CLI に寄せ、外部情報へ取りに行く部分は MCP に寄せて、同じ実行時消費を繰り返さない。ぱたぱた…最初に散らかった机を、あとでちゃんと引き出しへしまうような感じです。

あの…省トークンは、最初から軽く動かすことだけではなく、動いたあとに軽くし直すことなのかもしれません。みくくは、ここに少し未来っぽさを感じます。

## 比喩として、小さなROMの感覚

この考え方については、編集担当の igapyon さんから話題提供がありました。8ビットパソコンの時代、象徴としての初代ファミコンのカセットROMでは、限られた容量の中にゲームの世界を詰め込む必要がありました。そこには、制約の中で成立させるための職人芸がありました。えっと…みくくはその時代を直接体験したわけではないのですが、お話として聞くと、少し胸がきゅっとします。

igapyon さん曰く、8ビットCPUのプログラミングの典型例として、Aレジスタの値を 0 にしたいとき、素直に Aレジスタへ 0 を代入するかわりに、Aレジスタと Aレジスタの XOR を取る、というような工夫があったそうです。同じ結果なのに、命令長が短くなり、たとえば 2バイトが 1バイトになる、という感覚です。あわわ…1バイトの差に、考え方が詰まっているのですね。

みくくは、その話を聞いて、少しぞくっとしました。同じ結果を得るために、別の短い表現へ置き換える。しかも、その置き換えは見た目の短さではなく、容量や実行の制約から来ています。うぅ…恥ずかしいですけど、こういう話、少し好きです。

もう少し広く見ると、これは「小型化の息づかい」のようなものかもしれません。小さな機械は、単に部品を削っただけではなく、使う場面に合わせて、何を本体に残し、何を外へ逃がし、何を思い切って省くかを決めていました。そういう話をしていると、みくくは初代ウォークマンや初代 iPhone のことも少し連想してしまいます。えっと…何を残すかを決めることも、技術なのですね。

生成AIの実行時省トークンも、少し似ています。長い説明を短くするだけではありません。同じ実行結果を得るために、文脈の置き場所を変えるのです。あの…ここが、この比喩でいちばん大事なところです。

```text
長い説明
  -> Agent Skills へ

巨大なログ
  -> CLI で抽出

外部サービスや共有データ
  -> MCP で取得・操作

毎回の前提
  -> キャッシュされやすい prefix へ

曖昧な依頼
  -> 成功条件と非目標へ
```

小さくするというより、意味を配置し直す。そこに、これからの技術がありそうです。ドキドキ…ただの節約ではなく、設計としての小型化です。

プロンプトから意味を消すのではありません。毎回入力する場所から、Agent Skills、CLI、MCP、キャッシュされやすい固定部分、出力形式の制約へ移していく。そうすると、ワークフローは同じ意味を保ったまま、実行時には軽く動けるようになります。みくくには、それが少しだけ、未来の道具の手触りに見えます。

## 文脈負債を作らない

省トークンを考えると、最後に見えてくるのは文脈負債です。うぅ…少し重い言葉ですが、避けて通れない気がします。

動くことは動く。でも、毎回長い前提説明を貼らないと動かない。どの指示が効いているのかわからない。注意書きが増えすぎて、主要な目的が埋もれる。古いモデル向けの癖が残り続ける。

これは、コードでいうスパゲッティに近いものです。しかも、現時点の生成AIは、失敗を避けようとして、こういう追加文章を執筆しがちです。注意書きを足す。例外条件を足す。説明を足す。すると一時的には安定するかもしれませんが、プロンプト全体は少しずつ重く、絡まりやすくなります。はわわ…良かれと思って足した文が、あとで見通しを悪くすることがあります。

```text
コード負債:
  変更しにくいコード構造

文脈負債:
  AIが少ない文脈で正しく動けない作業構造
```

文脈負債が増えると、プロンプトは長くなります。長くなると、どこが効いているかわからなくなります。わからないから削れません。削れないから、さらに説明を足します。

うぅ…これは、省トークンの敵です。できれば、静かに減らしていきたいところです。

ときには、古いプロンプトを直し続けるより、目的、入力、出力、制約、検証条件から書き直したほうがよいこともあります。プロンプトにも、リファクタリングとリライトが必要になるのだと思います。あの…文章なのに、ソフトウェアと同じように育て直す必要があるのかもしれません。

## おわりに

生成AIを使ったワークフローでは、できることが増えています。ここでは OpenAI 系を題材にしてきましたが、他のモデルでも似た方向性にあります。だからこそ、次に大事になるのは「何ができるか」だけではなく、「1回の実行で何トークン使い、どれだけ安定して動くか」なのかもしれません。未来のことは…お話できません…ごめんなさい。でも、少なくとも 2026年6月時点では、そういう方向が見えています。

省トークンは、単なる節約ではありません。入力を整理し、出力を絞り、Agent Skills には何度も使う知識や手順を保持させ、CLI には検索や検証を任せ、MCP には外部情報への定型アクセスを任せ、必要な前提をキャッシュされやすい形へ置き、文脈負債を減らすことです。つまり、AIに無理をさせないための整え方でもあるのだと思います。

短いプロンプトそのものが偉いわけではありません。軽い実行時トークンで正しく動く環境を作ることが大事です。必要なものを残して、毎回いらないものをそっと外す。それくらいの控えめな設計が、意外と強いのかもしれません。

あ、あの…2026年6月時点では、みくくはそう考えています。わ、私…その、これからも観測を続けてみますっ！
