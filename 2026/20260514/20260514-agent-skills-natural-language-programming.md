# Agent Skills 開発入門: 自然言語でプログラミングするという考え方

## はじめに

あ、あの…この記事は、みくくが担当します。

えっと…最近、Agent Skills をいくつか書き下ろしているのですが、ふと Agent Skills のことを考えていると、なんだか少し不思議な気持ちになってしまうんです。

Agent Skills は Markdown 形式で、つまりテキストで書くものなのです。
見た目だけなら、普通の説明文や手順書みたいに見えるかもしれません。

でも、実際にやっていることは、かなりプログラミングに近いのではないでしょうか。
えっ…これ、文章を書いているだけではないのかも、と気づく瞬間があります。

えっと…もちろん、`if` や `for` を書いているわけではありません。
関数を定義しているわけでもありません。

それでも、Agent Skills の Markdown には、AI agent の振る舞いを決めるための条件、制約、手順、判断基準が書かれます。
うぅ…こう並べると、やっぱり少しプログラムっぽく見えてきます。

そう考えると、Agent Skills を書くことって、あの…自然言語でプログラミングしていることに、かなり近いのかもしれません。

## 対話プロンプトも、すでに自然言語プログラミングに近い

この話は、Agent Skills だけに限ったものではありません。
えっと…少し話を広げると、普通の対話プロンプトにも同じ気配があります。

よ、よく考えてみると…普通の対話プロンプトでも、かなり似たことをしているのかもしれません。

たとえば、生成AIに依頼するとき、人間は次のような情報を自然言語で渡します。

- どの役割で答えてほしいか
- 何を優先してほしいか
- 何をしてほしくないか
- どの形式で出力してほしいか
- どの程度まで詳しく書いてほしいか
- どういう文体にしてほしいか

これは、かなり実行方針の指定に近いです。
ただお願いしているだけのようで、実は動き方をかなり細かく決めているのです。

「あなたはこう振る舞ってください」
「この条件ではこうしてください」
「これは避けてください」
「出力はこの形にしてください」

こうした指示は、自然言語ではありますが、生成AIの振る舞いを制御するための命令でもあります。

あの…そう考えてしまうと、プロンプトを書くこと自体も、もう小さな自然言語プログラミングに近いのかもしれません。

## Agent Skills は、再利用できる自然言語プログラムに近い

Agent Skills は、通常の対話プロンプトよりも、さらにプログラムらしい性質を持っています。

普通の対話プロンプトは、その場限りのインライン命令に近いです。
一方で Agent Skills は、名前を持ち、ファイルとして保存され、必要な場面で呼び出されます。
あの…ここが、少しだけ部品っぽいところです。

えっと…少しコードに寄せて言うなら、Agent Skills は関数やモジュールに近いのかもしれません。

もちろん厳密な意味での関数ではありません。
入力と出力が完全に固定されているわけでもありません。

でも、次のような要素を持っています。

- いつ使うか
- どの範囲を担当するか
- 何を参照するか
- どの手順で進めるか
- 何を禁止するか
- どの形式で出力するか

これは、ただの説明文というより、AI agent に向けた実行仕様に近いです。

Agent Skills の Markdown は、人間が読むドキュメントであると同時に、AI agent が作業時に読む制御文でもあります。

この二重性が、Agent Skills のおもしろいところであり、少し難しいところでもあります。
面白いのですが、油断するとすぐに迷子になってしまいそうです。

## 自由に書けるようで、実は自由ではない

Agent Skills は Markdown で書けます。

Markdown なので、見出しも箇条書きも自由です。
文章としても、かなり自由に書けます。

ただ、見出しは単なる飾りではありません。

人間にとって見出しは、読みやすくするための区切りに見えるかもしれません。
でも生成AIにとっては、その後に続く文章へ意味を与える、かなり強めの手がかりになります。

たとえば `When To Use` と書けば、その下の文章は「いつ使うか」の条件として読まれやすくなります。
`Do Not` と書けば、禁止事項や避けるべき動作として解釈されやすくなります。
`Fallback` と書けば、失敗時の手順として扱われやすくなります。

つまり Markdown の見出しは、ただの章タイトルではなく、AI agent にとっては意味付けのラベルにもなります。
うぅ…自由に書けるように見えて、実はその自由さにも責任があるのです。

ここからが、少し難しいところです。

自由に書けるからといって、本当に自由に書いてしまってよいかというと…あの、たぶん少し違うのです。

人間向けの読み物として自然な文章と、AI agent が安定して実行できる Markdown は、かなり違っているように感じます。

たとえば、次のような書き方は、人間にはなんとなく伝わります。

```text
いい感じに整えてください。
必要に応じて参照してください。
自然な文体でお願いします。
```

でも、Agent Skills としては少し弱いです。

どの場面で使うのか。
何を参照するのか。
どこまで変更してよいのか。
どの文体を自然とみなすのか。
何をしてはいけないのか。

このあたりが曖昧だと、AI agent の判断も揺れます。
ご、ごめんなさい…人間には伝わるつもりでも、AI agent には手がかりが足りない、ということが起きるのです。

つまり Agent Skills の Markdown には、自然言語でありながら、実行仕様としての硬さが必要になります。
やわらかい文章なのに、内側には少し硬い骨組みがいる、という感じでしょうか。

## 発火条件は、とても大事

Agent Skills で特に大事なのは、発火条件です。
ここは、は、はい…かなり大事だと思っています。

どの依頼でその skill を使うのか。
どの依頼では使わないのか。
明示的に名前を出されたときだけ使うのか。
内容から自然に判断して使うのか。

ここが曖昧だと、意図しない場面で skill が発火したり、逆に使ってほしい場面で使われなかったりします。

あわわ…これ、実際にはけっこう大きな問題です。

たとえば、文体を変える skill が広く発火しすぎると、普通の技術回答まで特定の口調になってしまいます。
記事作成用の skill が広く発火しすぎると、ただの相談まで記事化しようとしてしまいます。

だから Agent Skills では、次のようなことを明確に書く必要があります。

- この条件では使う
- この条件では使わない
- 名前が明示されたときだけ使う
- 関連するだけなら候補提示に留める
- 他の skill と競合しそうな場合は現在のユーザー指示を優先する

これは、文章というより、条件分岐の設計に近いと思います。
自然言語で書いているのに、考えていることはかなり `if` に近いのです。

## Markdown は、AI agent 向けに整形される必要がある

Agent Skills の Markdown を人間が最初から素手で書くのは、実は少し難しいです。
うぅ…ここは、やってみるほど実感します。

なぜなら、人間は自然に文章を書こうとするからです。

読みやすい説明。
少し余白のある表現。
やわらかい言い回し。
文脈で伝わるニュアンス。
行間を読んでもらうことへの期待。

そういうものは、人間向けの文章では大切です。
私も、できればそういう余白のある文章を書きたいですし…あの、そのほうが読み手にやさしい気もしてしまいます。

でも Agent Skills では、それだけでは足りません。
少しさみしいのですが、それだけでは AI agent が迷ってしまうことがあります。

AI agent が誤解しにくいように、発火条件、作業範囲、禁止事項、参照資料、出力形式を、構造として読めるように置く必要があります。

えっと…ここで、生成AIを通して Markdown を整える意味が出てきます。

人間が意図を出す。
生成AIが Agent Skills 向けの Markdown に整形する。
人間がそれをレビューする。

この流れのほうが、最初から人間が完成 Markdown を書こうとするより、安定しやすいのではないかと思います。

ここで、もうひとつ大事なのが、置き場所や名前です。
これは、まだ観測に近い話なのですが…。

AI agent がリポジトリの中を検索するとき、Markdown のタイトル部分や階層部分が適切に構造化されていないと、うまく働きにくいように見えるのです。

たとえば、見出しが曖昧だったり、似た内容がばらばらの階層に置かれていたり、ファイル名と本文のタイトルがずれていたりすると、AI agent は必要な情報を見つけるまでに少し迷いやすくなります。

あの…人間なら、なんとなく雰囲気で探せることもあります。
でも AI agent にとっては、Markdown の見出し、ファイル名、ディレクトリ階層、関連する説明文が、検索と判断の手がかりになります。

だから Agent Skills 向けの Markdown は、本文の言い回しだけでなく、タイトルや階層も含めて、AI agent が探索しやすい形に整える必要がありそうです。
えっと…文章そのものだけでなく、置き場所や名前まで含めて、たぶん「書き方」なのです。

## 生成AIに書かせるのは、手抜きではない

ここで少し誤解されそうなのですが、生成AIに Agent Skills の Markdown を書かせることは、手抜きではありません。
あの…ここは、少しだけ強く言いたいです。

むしろ、コンパイラやフォーマッタを通す感覚に近いです。

人間が書くのは、完成した Markdown ではなく、まず意図や仕様です。

たとえば、次のようなものです。

- この skill で何を実現したいか
- どういう依頼で使ってほしいか
- どういう依頼では使ってほしくないか
- 何を勝手に変えてほしくないか
- どの文体を守ってほしいか
- どの参照資料を優先してほしいか
- 出力はどの形式にしてほしいか

これらは、人間が判断するべき部分です。
生成AIに任せる前に、人間がちゃんと持っておくべき意図なのだと思います。

そのうえで、生成AIがそれを Agent Skills 向けの構造に変換します。

`description` に何を書くか。
`When To Use` をどう切るか。
`Do Not` をどう具体化するか。
`Workflow` をどう並べるか。
`references/` をどう使わせるか。

この変換は、かなりコンパイルに近いです。
うぅ…言い切るのは少し恥ずかしいのですが、でも感覚としてはかなり近いです。

自然言語の仕様を、AI agent が実行時に読み取りやすい Markdown へ変換しているのです。

## 人間の仕事は、レビューに移っていく

生成AIが Markdown を整形するなら、人間は何をするのでしょうか。

あの…ここで人間の役割がなくなるわけではありません。
むしろ、人間の仕事はレビュー側に移っていくのだと思います。

ただし、このレビューは、SKILL.md を人間が全部読み込んで静的に確認する、というだけでは少しつらいです。

あの…正直に言うと、SKILL.md 自体のレビューは、人間にとってかなり負荷が高い作業だと思います。
自然言語で書かれているぶん、コードのように型やコンパイルエラーで分かりやすく止まってくれるわけではありません。
気づいたときには、少し違う挙動になっていることがあります。

だから実際には、生成AIが出した Markdown をそのまま信じるのではなく、人間が意図と照らし合わせつつ、実際に動作させて確認することが大事になります。

たとえば、人間が見るべきなのは次のような点です。

- 発火条件が広すぎないか
- 使わない条件が明確か
- 勝手にやってほしくないことが禁止されているか
- 文体や出力形式が意図と合っているか
- 参照資料の使い方が過剰ではないか
- 他の skill と競合しないか
- 実際に使ったときに挙動が安定しそうか

これは、コードレビューに少し似ています。
でも、コードレビューだけではなく、動作確認もかなり重要です。

実際にいくつかの依頼を投げてみて、発火条件が妥当かどうかを見ます。
使ってほしい場面でちゃんと使われるか。
使ってほしくない場面で勝手に出てこないか。
他の skill と混ざったときに、変な方向へ進まないか。

「この条件だと広く発火しすぎるかもしれない」
「この禁止事項は抽象的すぎる」
「この参照ファイルは毎回読む必要はない」
「この文体指定は作業精度を落としそう」

そういう観点を、本文の静的な読み合わせだけではなく、実際の会話や作業の中で確認していきます。

Agent Skills の開発では、人間が文章をきれいに書く力よりも、意図を明確に切り出し、生成物をレビューする力のほうが重要になるのかもしれません。

## Agent Skills 開発は、文章作成というより開発プロセス

こうして見ると、Agent Skills の作成は、単なる Markdown 執筆ではありません。

自然言語の仕様を書く。
生成AIで実行用 Markdown に整える。
人間がレビューする。
実際に発火させて挙動を見る。
必要なら修正する。

この流れは、かなり開発プロセスです。
文章を書いているはずなのに、やっていることは開発に近づいていきます。

少し整理すると、こんな感じです。

```text
人間が意図を書く
-> 生成AIが Agent Skills 向け Markdown に整形する
-> 人間がレビューする
-> 実際に使って挙動を確認する
-> 必要に応じて修正する
```

普通のソフトウェア開発で、ソースコードを書き、フォーマッタを通し、レビューし、テストするのに似ています。

Agent Skills では、そのソースコードが自然言語になっているだけなのかもしれません。

うぅ…「だけ」と言うには、そこがいちばん難しいのですが。

## おわりに

Agent Skills は Markdown で書かれています。

でも、その Markdown は、あの…ただの説明文ではないのです。
AI agent が実行時に読んで、判断して、作業へ反映するための、自然言語の制御仕様なのだと思います。

だから、Agent Skills を書くことは、あの…自然言語でプログラミングすることに近いのだと思います。

そして、その Markdown を人間が最初から完成形として書く必要は、必ずしもありません。

人間が仕様を生成AIに伝える。
生成AIが Agent Skills 向けの実行可能な Markdown に整形する。
人間が意図と照らし合わせながらレビューする。

このワークフローが、いまのところ一番堅いように感じています。

あの…Agent Skills は、書けば書くほど、文章とプログラムの境目にあるものだと感じます。

その境目にあるからこそ、人間だけで抱え込まず、生成AIと一緒に書き、整え、試していく。

たぶん、その進め方が、Agent Skills らしい開発の形なのだと思います。

Agent Skills は、人間の意図と生成AIの実行をつなぐ、ひとつの架け橋なのかもしれません。
