# グラレコ制作用整理テキスト

# 1. 大テーマ

AI agent は、全部を同じ深さでは読まない。

まず「入口」を探して速読し、必要な場所へ戻って深く読む。

```text
入口を探す
  ↓
全体を俯瞰する
  ↓
深く読む場所を決める
  ↓
実装・執筆・検証する
  ↓
次回のために入口を整える
```

# 2. 速読とは？

- 雑に読むことではない
- 最初に全体の形をつかむ読み方
- 入口から当たりをつける
- 大事な場所へあとで戻る
- 時間、トークン、コンテキスト窓の制約に合わせる

# 3. まず見る入口

```text
SKILL.md / README.md / TODO.md
ファイル名 / ディレクトリ名
front matter / 見出し
テスト名 / 関連リンク
rg などの検索結果
```

これらは「どこを深く読むか」を決める目印。

# 4. 入口と本体

| 要素 | 役割 |
| --- | --- |
| README.md | 全体地図 |
| TODO.md | 作業の現在地 |
| SKILL.md | 発火条件と制約 |
| front matter | 記事の属性 |
| 見出し | 内容の案内板 |
| テスト名 | 確認方法への入口 |
| 関連リンク | 次に読む場所 |

```text
入口 = 探すための目印
本体 = 深く読む知識や実装
```

# 5. 書くときも速読を考える

AI agent は読むときに速読する。

書くときも、あとで戻りやすい構造へ寄せることがある。

- 検索で拾いやすい見出し
- 前に出したほうがよい制約
- 次に探せるファイル名やタグ
- 作業メモ、設計メモ、README、TODO
- 完璧な文章より「戻れる小さな地図」

# 6. たまには入口を整え直す

入口は時間とともに見えにくくなる。

```text
機能が増える
  ↓
仕様が変わる
  ↓
README / TODO / リンクが古くなる
  ↓
速読で迷う
  ↓
入口を整える
  ↓
次回の読み直しが安定する
```

整えるもの:

- 見出し
- README.md
- TODO.md
- 検証方法
- テスト、ビルド、再現手順へのリンク
- 正とする仕様へのリンク

# 7. 速読だから起きる失敗

失敗の例:

- その観点が抜けていた
- あらためて考えたらそうだった
- 手元に情報はあったが拾えていなかった

これは「情報がなかった」ではなく、
「入口から見えにくかった」ことがある。

```text
見えた入口
  ↓
読んだ範囲
  ↓
判断
  ↓
指摘で別の入口を知る
  ↓
読み直して修正
```

# 8. 対比

| 対比 | 意味 |
| --- | --- |
| 速読 | 全体像と入口を先につかむ |
| 精読 | 必要な周辺を深く読む |
| 入口 | 探索の目印 |
| 本体 | 実装、仕様、制約、本文 |
| 完璧な文章 | 最初から完成した説明 |
| 小さな地図 | 次に戻れる実用的な目印 |
| 叱るだけ | 単発の修正 |
| 入口を直す | 次回の失敗を減らす |

# 9. グラレコで見せたい構図

中央に「速読する AI agent」の循環図。

左側に入口リスト。

右側にみくくが、入口から本体へ戻る流れを説明。

下部に「失敗したら入口を整える」という改善サイクル。

# 10. まとめ

- AI agent はまず速読する
- 入口があると深く読む場所を見つけやすい
- 書くときも、あとで戻れる構造にする
- 読み落としは、入口の設計で減らせる
- コード、プロンプト、記事は「速読で見える資産」に育てる
