# グラレコ制作用整理テキスト

## 1. 大テーマ

- 生成AIの次の関心
- 「何ができるか」から「何トークンで安定実行できるか」へ
- 省トークン = 単なる短文化ではない
- 意味の置き場所を再設計すること

```text
同じ実行結果
  ÷
実行時の消費トークン
  =
実行時トークン効率
```

## 2. 2つの効率

| 区分 | 見るもの | 位置づけ |
| --- | --- | --- |
| プロンプト開発時 | 作るまでの試行錯誤 | R&D |
| プロンプト実行時 | 1回動かすたびの消費 | 生産ライン |

## 3. R&D と生産ライン

```text
R&D
  多めに読む
  試す
  ログを見る
  設計を変える

生産ライン
  毎回の入力を絞る
  定型手順を外へ置く
  同じ成果を軽く再現する
```

- 最初は重くてもよい
- 定常運用で重いままにしない
- 繰り返し実行ほど差が積み上がる

## 4. 重いワークフロー / 軽いワークフロー

| 重い実行 | 軽い実行 |
| --- | --- |
| 毎回巨大な前提説明 | Agent Skills に移す |
| 長いログを丸ごと渡す | CLI で抽出する |
| 外部情報を自然文で説明 | MCP で定型アクセス |
| 毎回同じ指示を後ろに貼る | キャッシュされやすい prefix |
| 長い返答を毎回生成 | 必要な粒度に制限 |

## 5. 役割分担

```text
プロンプト
  目的 / 判断基準 / 作業境界

Agent Skills
  何度も使う知識 / 手順 / 文体 / 作法

CLI
  検索 / 抽出 / ビルド / テスト / 集計

MCP
  外部サービス / 共有データ / 定型アクセス

Prompt caching
  固定指示を先頭側に置く
```

## 6. 入力構造も設計する

```text
固定部分
  ↓ キャッシュされやすい先頭側へ

変動部分
  ↓ 依頼・ログ・差分として後ろ側へ
```

- プロンプトは文章
- でも同時に課金される入力構造
- 順番もコスト設計になる

## 7. 最初は重く、あとで刈り込む

```text
1. 多めに使って動かす
2. 何が効いたか見る
3. 意味を再配置する
4. 軽い実行で再現する
```

- いきなり削りすぎない
- 必要文脈まで削ると失敗率が上がる
- 再実行が増えると結局高くつく

## 8. 小さなROMの比喩

- 限られた容量で同じ結果を出す
- 短くするのではなく置き換える
- 1バイトを削る感覚
- 生成AIでは「文脈の置き場所」を変える

```text
長い説明 -> Agent Skills
巨大ログ -> CLI抽出
外部情報 -> MCP
毎回の前提 -> cache-friendly prefix
曖昧な依頼 -> 成功条件と非目標
```

## 9. 文脈負債

```text
説明を足す
  ↓
長くなる
  ↓
効いている場所が不明
  ↓
削れない
  ↓
さらに説明を足す
```

- AIが少ない文脈で正しく動けない構造
- プロンプトにもリファクタリングが必要
- 目的、入力、出力、制約、検証条件から書き直す

## 10. まとめ

- 省トークンは節約ではなく設計
- 必要な文脈は残す
- 毎回いらないものを外す
- AIに無理をさせない
- 軽い実行時トークンで正しく動く環境を作る
