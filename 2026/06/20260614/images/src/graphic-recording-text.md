# グラレコ制作用整理テキスト

## 1. 大テーマ

**生成AI時代の既存システム文書化**

設計書ゼロの既存システムを、AI agent だけで一気に文書化するのではなく、ソフトウェア考古学のように痕跡を読み、仮説を作り、有識者レビューで確かめながら、実務で使える仕様メモへ蒸留する。

```text
痕跡を拾う
  ↓
仮説を作る
  ↓
有識者レビュー
  ↓
仕様メモへ蒸留
  ↓
次の調査・改修に使える
```

## 2. なぜ一気に生成しないのか

### 対比

| いきなり自動生成 | ソフトウェア考古学 |
| --- | --- |
| ソース全体を一気に読む | 小さな入口から読む |
| 情報量が多すぎる | 見取り図から始める |
| 業務意味が抜けやすい | 有識者の知識で補う |
| 完成設計書を期待する | 育つ仕様メモを作る |

キーワード:

- ソースコードは事実を持つ
- 業務上の意味はコードだけでは確定しにくい
- 人間が理解しやすい順番に並べ直す

## 3. 最初の入口

まず作るのは、完成した設計書ではなく **一覧系**。

```text
画面一覧
バッチ一覧
帳票一覧
API一覧
```

役割:

| 一覧 | 手がかり |
| --- | --- |
| 画面一覧 | Controller, View, 画面ID, メニュー定義 |
| バッチ一覧 | main クラス, ジョブ定義, シェル |
| 帳票一覧 | 帳票ID, テンプレート, PDF/Excel 出力 |
| API一覧 | Controller, endpoint, HTTP method |

意味:

- 人間が見られる入口
- AI agent の調査範囲を絞る足場
- 有識者レビューの叩き台

## 4. テーブル設計は古い地図

DB を使う業務システムでは、テーブル設計が強い手がかりになる。

```text
物理名だけ
  ↓ 読みにくい
論理名・項目説明あり
  ↓
業務上の意味として読める
```

見たい情報:

- テーブル名 / カラム名
- 型 / 桁 / NULL / 主キー / インデックス
- 論理名
- 項目説明
- コード値の意味

ポイント:

- 古くても現行 DB と照合できる
- 物理名と論理名の対応で AI agent の理解が進む
- データ操作が業務処理として読めるようになる

## 5. 埋もれやすいもの

### 帳票

帳票はテンプレートだけでは見つからないことがある。

例:

- Web 画面を PDF 出力
- CSV を Excel 加工して帳票化
- 画面内容を Excel に貼り付ける運用
- 業務上は帳票だが、コード上は帳票に見えないもの

### 手ごわい一覧

| 種類 | 難しさ |
| --- | --- |
| ジョブ一覧 | スケジューラ、手動実行、障害時実行が混ざる |
| 外部連携一覧 | URL、ファイル、キュー、環境変数、運用メモに分散 |

外部連携は、システム外の人間作業や Excel 加工まで仕様になっていることがある。

## 6. 一覧は育てるもの

最初の一覧は仮でよい。個別仕様を抽出するたびに、一覧の解像度が上がる。

```text
仮の一覧
  ↓
画面仕様 / バッチ仕様 / 帳票仕様 / API仕様
  ↓
一覧を更新
  ↓
機能一覧の輪郭
```

抽出する仕様:

- 画面: 入力、表示、検索条件、検証、更新、遷移、権限
- バッチ: 起動条件、対象データ、処理、更新先、ログ、異常時
- 帳票: 出力条件、項目、集計単位、並び順
- API: request, response, 認証、エラー、参照・更新対象

## 7. 機能一覧は深い地層

機能はコード上にそのまま現れない。

```text
画面・バッチ・帳票・API
  ↓ 材料がそろう
機能の輪郭が見える
  ↓
不完全な機能一覧
  ↓
有識者ヒアリングがしやすくなる
```

注意:

- 廃止機能は大きなリスク
- AI agent の補完は「推定」と札を付ける
- 完全でなくても叩き台として価値がある

## 8. 有識者レビューで照合する

AI agent だけで完結させない。

```text
AI agent が作る
  ↓
有識者が見る
  ↓
抜け・誤り・業務意味・運用注意を補う
  ↓
AI agent が反映
  ↓
再解析
```

レビューで得たいもの:

- その処理の業務上の意味
- 今も使っているか
- 廃止・例外・運用手順
- ソースコードには出ない前提

## 9. コメント・ログ・メモも遺物

正式設計書ではなくても、文脈復元の材料になる。

| 遺物 | 役割 |
| --- | --- |
| コメント | 当時の判断や迷い |
| Git ログ | 変更理由の断片 |
| リリースノート | 業務イベントとの接点 |
| 運用メモ | ソース外の仕様 |
| Excel | 現場の理解、手作業の正本 |

## 10. 仕様メモへ蒸留する

集めた断片を、きれいな物語ではなく、実務で使える仕様メモへ濃縮する。

分けて書く:

- 確認できたこと
- 推定していること
- 矛盾していること
- 有識者に確認したいこと

最終イメージ:

```text
ソースコードの事実
+ DB / 設定 / ログ / コメント
+ 運用メモ / Excel
+ 有識者の記憶
  ↓
確認と推定を分ける
  ↓
改修・不具合調査に使える仕様メモ
```

## 11. グラレコ中央メッセージ

**完成設計書を一度で生成するのではなく、痕跡から仕様メモを育てる。**

吹き出し案:

- 「あ、あの…まずは小さな一覧から始めます…！」
- 「コードは事実、でも業務意味は照合が必要なのです…」
- 「推定には札をつけて、レビューで育てます…」
- 「最後は、次の人が使える仕様メモへ蒸留します…」
