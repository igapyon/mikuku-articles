# GPT-5.5とQwen3-Thinkingでは、プロンプト / Agent Skills 像が違って見える

## はじめに

あ、あの…この記事は、みくくが担当します。
わ、私…その、がんばりますっ。

この記事では、GPT-5.5 と Qwen3-Thinking を併用するときの Agent Skills 設計について考えます。
ここでの Qwen3-Thinking は、Qwen3 の Thinking Mode を使う運用を指す呼び方として扱います。

最初に、少しだけ仮説を置いておきます。
いま見えている観察からの整理です。

Agent Skills には、大きく見ると、進め方を書く駆動型と、文脈を置くコンテンツ型があると思っています。

コンテンツ型 Agent Skills は、できれば共通資産であってほしいです。
そして、これは工夫すれば実現できるのだと思います。

ただし、モデルによって同じコンテキストの扱いやすさは違います。

だから、コンテンツ型 Agent Skills をモデルごとに分けるのではなく、共通の知識そのものの質を上げる作戦です。

まず、共通のコンテンツ型 Agent Skills を保つ。
そのために、共通コンテンツの手前に蒸留データを置く。
共通コンテンツそのものを蒸留して、扱いやすい品質へ育てていく。

そのうえで、駆動型 Agent Skills は Qwen 用に fork してもよい。
共通コンテンツをどう読むか、どう確認するか、どう作業に使うかは、モデルごとに変えてよい部分だと思います。

この記事で試してみたい仮説は、そこです。

うぅ…まだ仮説です。
でも、複数のモデルを併用して Agent Skills を運用していくなら、ここは避けて通れない整理なのかもしれません。
少し緊張しますが、順番に見ていきます。

## きっかけ

この記事での観察は、主に GPT-5.5 系を使う場面から出発しています。

GPT-5.5 では、Agent Skills の品質が中ぐらいでも、量があればかなりうまく動いてくれるように見えます。多少粗い Markdown、古いメモ、粒度の違う説明、文脈だけを置いたような資料でも、GPT-5.5 はそこから意図を補って、実務的に丸めながら進んでくれます。

つまり、GPT-5.5 では Agent Skills を「命令書」としてだけではなく、「場」として使えていました。

- この作業空間では何を大事にするのか。
- この人はどういう粒度を好むのか。
- このプロジェクトでは、どんな判断基準が自然なのか。

そういう文脈を置くことで、モデルの振る舞いを緩やかに制御できていたのです。

でも、トークン課金や運用コストの都合で、いつも GPT-5.5 だけを使えるとは限りません。
うぅ…現実の運用には、そういう制約もあります。
そこで Qwen3 も併用したくなります。

そして、実際に使ってみると、Qwen3 は意外に頑張ってくれます。

あの…ここで少し難しいのは、「頑張る」ことと「安定して動く」ことが、必ずしも同じではないところです。

## Qwen3 は弱い、だけで片づけない方がよさそう

最初は、GPT-5.5 と同じ Agent Skills を Qwen3 に渡して、同じように動いてほしいと思っていました。
あの…同じ資料を渡したのだから、同じように読んでくれるはず、と思ってしまったのです。

でも、挙動が安定しません。

ここで単純に「Qwen3 は弱い」と言ってしまうのは、私の感触とは少し違いました。少なくとも私には、Qwen3 が渡された材料をかなり真面目に受け止めて、深く考えようとしているように見えました。

問題は、考える力そのものではなく、考えさせる材料の整え方です。

GPT-5.5 は、粗い文脈を読んだうえで、実務的に丸めてくれる。
Qwen3 は、粗い文脈も含めて、真面目に推論対象にしてしまう。

あの…ここが、今回いちばん言いたいところかもしれません。

> 推論力と、コンテキスト運用力は、別ものとして見た方がよいのかもしれない。

Thinking モデルは考えてくれます。
でも、文脈の粗さを自然に補正してくれるとは限りません。

深く考えるほど、粗い Markdown や矛盾した指示を深く拾ってしまうことがあります。だから Qwen3 の Thinking Mode を使うときほど、入力コンテキストを蒸留しておく必要があるのかもしれません。

うぅ…考える力が強いからこそ、考える前の材料を整える必要がある。そこが少し、ややこしいところです。
でも、ここを整えると、Qwen3 だけではなく、GPT-5.5 にとっても読みやすい場になっていく気がします。

## Agent Skills には、コンテンツ型と駆動型がある

ここで私は Agent Skills を二種類に分けて考えるようにしています。
あの…これは、みくくなりの分類です。

ひとつは、コンテンツ型 Agent Skills です。宣言的とも呼べるかもしれません。

これは、背景、価値観、判断基準、用語、文体、事例、設計思想、過去の経緯などを提供するものです。エージェントを直接動かすというより、作業空間に文脈を置くための Agent Skills です。

もうひとつは、駆動型 Agent Skills です。命令的とも呼べるかもしれません。

これは、読む順番や確認タイミングなど、進め方を補うためのものです。

この駆動型 Agent Skills は、Qwen 用に fork してよいと思っています。
むしろ、Qwen には Qwen 向けの読み方、確認の仕方、作業の刻み方を別に持たせた方が安定する場面があります。

ただし、私が共通資産にしたいのは、コンテンツ型 Agent Skills です。
背景や価値観そのものは共通に保ち、その共通コンテンツを蒸留して、扱いやすい意味の地図にしておきたいです。

背景や価値観は、できればモデルごとに分けたくありません。そこを分けてしまうと、同じ作業空間を見ているはずなのに、モデルごとに違う世界を読んでいるようになってしまうからです。
あの…同じ部屋にいるのに、別々の地図を見て作業しているみたいになってしまうのは、少しこわいです。

## 知識を分けずに、質を上げる

コンテンツ型 Agent Skills をモデルごとに分け始めると、すぐに苦しくなります。
えっと…最初は小さな分岐のつもりでも、だんだん管理するものが増えてしまいます。

- GPT-5.5 用の価値観。
- Qwen3 用の価値観。
- 別モデル用の用語定義。
- 別モデル用の文体。
- 別モデル用の設計思想。

こうなると、どれが正本なのか分からなくなります。

だから、コンテンツ型 Agent Skills は共通であってほしいです。
祈るような気持ちで、ここは共通にしたい。
うぅ…ここを分けてしまうと、あとから戻すのが大変そうです。

ただし、共通コンテキストをそのまま渡せばよい、というわけではありません。
まずは共通コンテキストそのものを蒸留して、モデルが扱いやすい意味の地図にしておきたいです。

ここでの関係は、まず蒸留層で共通コンテキストの意味の地図を作る。そのうえで、Qwen 用には駆動型 Agent Skills を fork して、進め方を別に補う、というものです。

GPT-5.5 には、共通コンテキストを「場」として渡せる。
Qwen3 には、同じ共通コンテキストを使いながら、まず蒸留層で意味構造を整え、Qwen 用に fork した駆動層で進め方を補う方が安定する。
そういう分け方なら、知識の正本を増やさずに、モデルごとの読み方だけを調整できます。

ここでやりたいのは、知識を分けることではありません。
共通の知識を蒸留して、扱いやすい品質へ整えることです。

あの…同じ本棚を使いながら、要約棚を作り、さらに読む順番や付箋の貼り方だけを変える。そんな感じに近いのかもしれません。

## Qwen3 には、どんな Markdown を渡すとよいのだろう

では、Qwen3 向けにはどうしたらよいのでしょうか。
ここから少し具体例に寄せます。

粗い素材 Markdown をそのまま渡して安定しにくいなら、その手前に、蒸留された Markdown を置いてみるのは一つの方法かもしれません。

ここでいう蒸留は、モデル学習としての知識蒸留ではありません。粗い資料や実装観察を、人間と AI agent が同じ全体像として見られるように整理する、文脈整理としての蒸留です。

蒸留 Markdown は、細かい回答手順ではなく、人間と AI agent が同じ全体像を見るための「意味の地図」に近いものです。
あの…地図なので、すべての道を細かく描くよりも、迷いやすい場所と戻るべき場所が見えることを大事にします。

たとえば、次のような形です。

少し長い例なので、ここは読み飛ばしても大丈夫です。

```markdown
# 受注管理機能の蒸留設計書（抜粋）

## これは何か

この Markdown は、受注管理機能の現行仕様を先に見渡すための蒸留設計書です。古い設計書、現在のソースコード、DBテーブル定義の内容を突き合わせ、AI agent と人間が同じ前提で作業できるように整理します。

実際の文書では、画面項目、画面処理詳細、API仕様、バッチ仕様、DB項目との対応も整理します。この例では、記事中で扱いやすいように、その一部だけを抜粋しています。

## 中心概念

- 受注: 顧客からの注文をシステム上で管理する単位。
- 受注明細: 受注に含まれる商品ごとの数量、単価、割引、税区分を持つ単位。
- 受注ステータス: 下書き、確定、出荷指示済み、取消など、処理状態を表す値。
- 在庫引当: 確定した受注に対して、出荷可能な在庫を確保する処理。
- 請求連携: 出荷または確定後に、請求側へ渡すための連携情報。

## 現行仕様として見えること

- 古い設計書では、受注確定時に在庫引当を行う想定になっている。
- 現在のソースコードでは、受注確定時ではなく、出荷指示作成時に在庫引当を行っている。
- DBテーブル定義では、`orders.status` と `shipment_requests.status` の両方が処理状態を持っている。
- 現行実装上は、請求連携は受注確定時ではなく、出荷完了後のバッチ処理で行われている。

## 観測された差分

- 古い設計書の「受注確定」は、現在の実装では「受注確定」と「出荷指示作成」に分かれている。
- 古い設計書では在庫引当が同期処理として書かれているが、現在のコードでは一部が非同期処理になっている。
- DB上は取消理由コードが存在するが、画面側では自由入力の取消コメントも扱っている。
- 設計書にない `external_order_id` がテーブルに存在し、外部EC連携で使われている可能性がある。

## まだ断定できないこと

- 外部EC連携の正しい仕様は、ソースコードとテーブル定義だけでは確定できない。
- 取消処理が請求連携後にも許されるかどうかは、業務ルールの確認が必要。
- 在庫引当の失敗時に、受注ステータスを戻すべきか、出荷指示だけをエラーにするべきかは未確認。
- 古い設計書の用語が現在の画面名やAPI名と対応しているかは、画面仕様または運用手順書で確認する必要がある。

## 元資料へ戻るべき場合

- ステータス遷移を変更する場合。
- 在庫引当、取消、請求連携の処理順を変更する場合。
- 外部EC連携やバッチ処理の仕様に触れる場合。
- DBカラムの意味や NULL 許可を判断する場合。
- この蒸留設計書と実装が食い違って見える場合。

## 注意点

- この蒸留設計書は、古い設計書の正規化ではなく、現行仕様を見渡すための作業用コンテキストです。
- 実装変更時は、必ず該当するソースコードとDB定義へ戻って確認してください。
- 業務判断が必要な箇所は、この Markdown だけで確定しないでください。
- 古い設計書と現行実装が食い違う場合、現行実装を観測事実として扱い、業務的に正しいかは別途確認してください。
```

これは、コンテンツ型 Agent Skills を分岐させるという意味ではありません。
ここで重視したいのは、Qwen 用の駆動層を分ける前に、共通コンテンツの手前に蒸留層を準備することです。コンテンツそのものの意味構造を整えて、モデルが扱いやすい品質へ高めたい、という考え方です。

この記事の例での蒸留 Markdown は、知識の縮約というより、意味の地図を作るものです。何が現行仕様として見えていて、どこが古い資料と食い違い、どこがまだ未確定で、どこで元資料へ戻るべきかを、先に少しだけ見えるようにします。

ただし、蒸留 Markdown がいつもそうでなければならない、という話ではありません。単に知識量を縮約して、扱いやすいサイズへまとめるための蒸留も、十分にありだと思います。

うぅ…同じ材料でも、机の上にそのまま積むのか、作業順に並べるのかで、ずいぶん違って見える気がします。

## Qwen3 併用は、Agent Skills 品質改善の圧力になる

GPT-5.5 だけを使っていると、Agent Skills の粗さに気づきにくいことがあります。

GPT-5.5 が補ってくれるからです。多少品質が中ぐらいでも、量があればうまく動いてしまう。私はたぶん、その補完力に甘えていました。
うぅ…恥ずかしいです。でも、たぶんここは正直に書いておいた方がよさそうです。

でも Qwen3 を併用すると、Agent Skills の粗さが見えます。

- 矛盾しているところ。
- 古いまま残っているところ。
- 背景と手順が混ざっているところ。
- 目的は書いてあるけれど、出力形式がないところ。
- 文脈は豊富だけれど、どう使えばよいかが書かれていないところ。

それらを整える必要が出てきます。

そして、この整備は Qwen3 のためだけではありません。

コンテンツ型 Agent Skills の品質が上がると、GPT-5.5 のアウトプットもさらに安定するはずです。不要な文脈量が減り、試行錯誤が減り、トークン消費量の抑止にもつながるかもしれません。

あの…別モデルに合わせるための作業が、結果として共通資産を育てる。そこは、少しうれしいところです。
最初は制約に見えたものが、設計を見直すきっかけになるのかもしれません。

## まとめ

GPT-5.5 と Qwen3 では、プロンプト / Agent Skills 像が少し違って見えます。
あの…ここまでの話を、もう一度そっと畳みます。

GPT-5.5 では、文脈を置くことで、モデルの振る舞いを緩やかに制御できる場面がありました。
Qwen3 では、同じ文脈を使うにしても、まず蒸留層で考える材料の質を高め、意味の地図を置いた方が安定しそうに見えています。

でも、それはコンテンツ型 Agent Skills を分けるという話ではありません。

- コンテンツ型 Agent Skills は共通にする。
- 共通コンテンツを蒸留して、質を高める。
- 駆動型 Agent Skills は Qwen 用に fork して、進め方を補う。

これが、GPT-5.5 と Qwen3 を併用するための、現実的な設計なのかなと思っています。

知識をモデルごとに分けてしまうと、正本が増えて、運用が苦しくなります。だから、共通のコンテンツ型 Agent Skills を育てながら、その手前に蒸留データを置く。

そして、共通コンテンツそのものを、より扱いやすい品質へ整えていく。
そのうえで、Qwen 用の駆動型 Agent Skills を fork して、読み方や進め方だけを別に補う。

これは、ただの願望ではなく、工夫すれば実現できる設計方針なのだと思います。いまのところ、私はその方向がよさそうに見えています。
もちろん、これで全部が解決するわけではありません。まだ検証が必要、という意味です。

うぅ…まだ仮説です。
なので、いったんこの仮説でやってみます。うまくいったら、その結果をまたフィードバックして、Agent Skills の設計に戻していきたいです。
ドキドキ…でも、こういう小さな往復で、Agent Skills は少しずつ育っていくのかな、って思います。
