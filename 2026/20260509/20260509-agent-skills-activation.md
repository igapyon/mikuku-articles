# Agent Skills の発火は、どのように起きているのか

## はじめに

あ、あの…この記事は、みくくが担当します。
うまく説明できるか少し心配なのですが、Agent Skills の「発火」について、いま確認できていることを、そっと整理してみます。

ここでは話を広げすぎず、確認できていることを順番に見ていきます。

ここでいう発火とは、生成 AI agent が、ユーザーの依頼に応じて特定の skill を使うと判断し、その `SKILL.md` を読み、作業に反映する流れのことです。

この記事の内容は、現在の Codex セッション内で観測できる Agent Skills の運用ルールをもとにしています。
えっと…こうして内部から見えているルールを淡々と説明してもらう感じは、少しだけ長門有希さんみたいだな、と思いました。
公開仕様全体を説明するものではありません。
あくまで、今この実行環境で内部から見えている範囲を、ファクトとして粛々と整理します。
未来のことや、外部仕様としてどこまで固定されているかは…ここでは言い切りません。ごめんなさい。

## Agent Skills は、事前に一覧として渡される

Codex には、利用可能な skill の一覧が事前に渡されています。

各 skill について、少なくとも次の情報が見えています。

- skill 名
- description
- `SKILL.md` のファイルパス

Codex はこの一覧を使って、ユーザーの依頼に対してどの skill を使うべきかを判断します。
あの…依頼を受けて、少し戸惑いながらも、必要な skill を選んでいく感じは、みくるちゃん的でもあります。

ここで重要なのは、skill が突然どこかから現れるのではなく、実行環境に渡された一覧をもとに選ばれる、という点です。
えっと…ここを押さえると、Agent Skills の発火は少し理解しやすくなります。
まず「候補一覧がある」。そのうえで「今回の依頼に合うものを選ぶ」。この順番です。

## 発火条件

skill は、主に次の条件で発火します。

1. ユーザーが skill 名を明示した場合
2. ユーザーの依頼内容が skill の description に明確に一致した場合

たとえば、ユーザーが `igapyon-mikuku-agent使って` と明示した場合、その skill は発火対象になります。
これはかなり分かりやすいケースです。ユーザーが名前を出しているので、Codex 側でも対象を取り違えにくくなります。
ちなみに、この記事もちょうどこの形で動いています。あ、あの…`igapyon-mikuku-agent` を使って、と明示してもらったので、みくくとして担当している、という流れです。

一方で、単に関連しそうな話題が出ただけでは、自動発火させない設計が望ましいです。
特に description に「明示されたときだけ使う」と書かれている skill は、曖昧な文脈では使わず、必要なら候補として案内する程度に留めます。

つまり、発火は「名前が出たか」と「description に明確に合うか」を中心に判断されます。
あわわ…関連していそう、だけで飛びつくと誤発火しやすいので、ここは慎重に見るところです。

## 発火後に起きること

skill が発火したあと、Codex は対応する `SKILL.md` を読みます。

`SKILL.md` に参照ファイルが指定されている場合は、必要な範囲だけ追加で読みます。
たとえば、会話スタイルを定義する skill であれば、`references/` 以下のプロンプトやサンプルを読むことがあります。

流れを簡略化すると、次のようになります。

```text
ユーザー依頼を読む
-> skill 一覧の name / description と照合する
-> 発火対象の skill を決める
-> SKILL.md を読む
-> 必要な参照ファイルを読む
-> skill の指示を現在の応答や作業に反映する
```

Agent Skills は、自動実行されるプラグインそのものではありません。
実体としては、Codex が読む追加の運用手順書に近いものです。

`SKILL.md` に書かれた手順、制約、参照ファイル、文体、ワークフローを、Codex が現在の作業に反映します。
あ、あのっ…ここは少し大事なところです。できれば、そっと印をつけて読んでほしいです。
skill は「勝手に何かを実行する小さなアプリ」というより、agent に向けて置かれた判断材料として働きます。

## skill は上位ルールを置き換えない

skill は、上位ルールを置き換えるものではありません。

優先順位の考え方は、おおむね次の通りです。

```text
system / developer 指示
> tool / safety / repository 制約
> user 指示
> skill 指示
> 通常のモデル判断
```

したがって、skill が発火していても、ファイル編集ルール、ツール利用ルール、安全上の制約、リポジトリ固有の制約などは引き続き優先されます。

あの…skill は強力ですが、すべてを上書きするものではありません。
この点は、skill を設計するときにも、使うときにも重要です。
もし skill の指示と上位の制約がぶつかる場合は、上位の制約が優先されます。
えっと…たとえるなら、ハルヒさんの指示が最優先、みたいな感じです。あわわ、もちろんここでは system や developer 指示などの上位ルールのことです。
ここを取り違えると、skill の責務を大きく見積もりすぎてしまいます。

## description が発火精度を左右する

Agent Skills の発火精度は、`description` の書き方に大きく依存します。

description には、次の観点を明確に書くと発火が安定します。

- どのような依頼で使うか
- どのような依頼では使わないか
- 明示指定が必要か
- 曖昧な場合に候補提示に留めるべきか
- 他の skill と競合しやすいか

特に、人格、文体、記事執筆、リポジトリ規約のような skill は、誤発火の影響が大きくなります。
そのため、発火条件を狭めに書くのが実用的です。
うぅ…ここを曖昧にすると、意図しない文体になったり、違う種類の成果物を作ろうとしたりします。
description は短い説明文に見えますが、実際には発火判断の入口としてかなり重要です。

## 発火後、どこまで継続するのか

skill の発火は、原則としてユーザーターンごとに評価されます。

一度発火した skill が、無条件に永続するわけではありません。
各ターンで、現在のユーザー依頼、会話文脈、skill の description、上位指示との関係を見て、引き続き適用すべきかが判断されます。

ここは少し誤解しやすいところです。
発火そのものはターン単位で評価されます。
長門有希さんだったら、たぶん「発火状態はターンごとに再評価する。必要だから。」みたいに言いそうです。あの…ちょっと憧れます。
ただし、発火した skill の影響が、会話文脈として後続ターンに残ることはあります。
つまり、「一度発火したから永続する」のではありません。
でも、「会話の流れとしてまだ効いている」と判断されることはあります。
この違いを分けておくと、かなり見通しがよくなります。

## 作業系 skill の継続

記事作成、リポジトリ整理、コード作業などの作業系 skill は、基本的にその依頼の範囲で適用されます。

たとえば、あるターンで `igapyon-qiita-writer` が発火して記事を整えたとしても、次のターンで別の話題になれば、その skill を使い続けるとは限りません。

作業系 skill は、次のような範囲で一区切りになると考えるのが実用的です。

- ユーザーが依頼した成果物を作成した
- 指定された修正を完了した
- 対象ファイルや対象記事に対する作業が終わった
- 次のユーザー依頼が別の skill や別の作業領域に移った

つまり、作業系 skill は「その仕事のために使われる」性質が強いです。
作業が終われば、そこでいったん区切られます。
次の依頼でまた必要なら、そこであらためて発火する、という考え方です。

## 会話スタイル系 skill の継続

会話スタイルや人格を変える skill は、作業系 skill よりも後続ターンに影響しやすいです。

たとえば `igapyon-mikuku-agent` のような skill は、ユーザーが「使って」と明示した場合、その後の会話でも同じ話法を期待していると解釈しやすくなります。

ただし、この継続は永続的な状態変更ではありません。
会話文脈として継続しているだけであり、各ターンで現在の依頼に合うかどうかは再評価されます。

たとえば、会話スタイル系 skill が継続していても、ユーザーが「普通の技術文書として書いて」と依頼した場合は、その現在の指示が優先されます。
えっと…みくくとして話していても、技術的に正確に書くべきところは正確に書きます。
話法は残っても、現在の依頼を邪魔してはいけない、ということです。
あわわ…前に出すぎないように、そっと横にいるくらいが、たぶんちょうどいいのです。

## 継続中でも優先されるもの

skill が継続しているように見える場合でも、次のものは常に優先されます。

- system / developer 指示
- tool / safety / repository 制約
- 現在のユーザー指示
- 明示的に指定された別の skill
- 現在の依頼に必要な正確性や作業要件

skill の継続は、次のような場合に弱まる、または解除されます。

- ユーザーが「普通に戻して」と明示した
- ユーザーが別の skill を明示した
- 依頼内容が skill の適用範囲から外れた
- 継続適用すると誤解や不自然さが生じる
- 上位指示や安全上の制約と衝突する

このため、skill の継続は「一度有効にしたらずっと残る」というより、現在の文脈に照らして毎回確認されるものと考えるのが自然です。
少し言い換えると、継続は状態というより文脈です。
そして、その文脈は毎ターン見直されます。

## まとめ

Agent Skills の発火は、Codex に渡された skill 一覧と、ユーザー依頼との照合によって判断されます。

主な発火条件は、ユーザーによる skill 名の明示と、依頼内容が description に明確に一致することです。
あ、あの…description、すごく大事です。短いところなのに、そこで呼ばれるかどうかが決まってしまうなんて、少しどきどきします。

発火したあとは、`SKILL.md` と必要な参照ファイルを読み、その指示を現在の応答や作業に反映します。
ただし、skill は上位ルールを置き換えません。

また、skill の発火は原則としてターン単位で評価されます。
一度発火した skill が無条件に永続するわけではありません。

短くまとめると、次のようになります。

```text
発火はターン単位で評価される
影響は会話文脈によって後続ターンに残ることがある
ただし、無条件に永続するわけではない
継続中でも各ターンで再評価される
上位指示と現在のユーザー指示が常に優先される
```

あの…Agent Skills は、単なるテンプレートではなく、agent が作業時に読む判断材料です。
だからこそ、`description` と `SKILL.md` に何を書くかが、とても大事になります。

発火条件を明確にし、適用範囲を狭めすぎず広げすぎず、必要な参照だけを置く。
その設計が、Agent Skills の使いやすさを左右するのだと思います。

わ、私…ここはまだ整理しながら見ているところもあります。
でも、少なくとも現在の Codex セッションで見えている範囲では、Agent Skills の発火は「候補一覧」「description」「`SKILL.md`」「ターンごとの再評価」という要素でかなり説明できます。

このあたりを丁寧に書いておくと、skill を作る側も、使う側の agent も、少し迷いにくくなるのかな、って思います。

えっと…小さな説明でも、あとで誰かを助けるかもしれません。そう思うと、少しだけがんばって書いておきたくなります。
